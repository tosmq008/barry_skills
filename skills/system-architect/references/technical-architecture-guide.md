# 工程架构与技术架构设计指南

> 本指南服务于 `SKILL.md` 的 **Phase 4（工程架构设计）** 和 **Phase 5（技术架构与核心选型）**。
> 它的职责不是直接推荐某个框架，而是把 Phase 3 得到的抽象模块先落到真实工程结构，再说明系统如何运行、代码如何分层、能力如何复用以及如何扩展。

## 1. 使用边界

### Phase 4 要解决的问题

- 抽象模块如何落到服务、应用、包、目录、仓库或 workspace
- 模块职责、依赖方向、共享能力、扩展点如何定义
- 存量系统哪些部分能复用，哪些需要重构或拆分

### Phase 5 要解决的问题

- 系统整体如何运行
- 代码如何分层
- 核心能力如何复用且保持高内聚
- 新需求以后如何低成本扩展
- 关键技术方案为什么这样选

### 本指南的核心输出

- 工程模块设计说明
- 模块依赖与边界说明
- 运行链路设计
- 代码分层设计
- 扩展与复用策略
- 技术选型评估矩阵
- ADR 记录

## 2. Phase 4：工程架构设计

### 2.1 从抽象模块落到工程结构

工程架构不是再做一次概念抽象，而是回答“代码最终怎么组织”。

常见落地单元包括：

- 服务
- 应用
- 包 / module
- 目录
- mono-repo workspace
- 共享库 / SDK
- 基础设施组件

推荐按以下顺序推导：

```text
抽象模块 -> 发布边界 -> 部署边界 -> 代码边界 -> 依赖边界 -> 共享边界
```

### 2.2 工程模块设计画布

每个工程模块至少描述以下内容：

| 字段 | 说明 |
|------|------|
| 模块名 | 该模块在工程中的唯一名称 |
| 来源模块 | 对应的抽象模块 |
| 主要职责 | 该模块必须承担什么，不承担什么 |
| 对外接口 | 对外暴露的 API、事件、命令、组件或契约 |
| 依赖模块 | 允许依赖哪些模块 |
| 数据归属 | 核心数据或资源由谁拥有 |
| 运行边界 | 是否独立进程、独立部署、独立发布 |
| 复用方式 | 被其他模块复用的稳定面 |
| 扩展点 | 未来新增能力如何接入 |

### 2.3 依赖方向规则

- 业务模块可以依赖平台能力，平台能力不能反向依赖具体业务
- 上层编排可以依赖下层能力，下层不依赖上层流程
- 共享库只沉淀稳定能力，不沉淀临时业务逻辑
- 跨模块访问优先走公开接口，不直接引用对方内部实现
- 如果两个模块必须频繁双向调用，优先重新检查边界是否划错

### 2.4 共享能力放置原则

不要把所有公共代码都塞进 `common` / `utils`。

可以下沉为共享能力的内容：

- 稳定协议与 SDK
- 认证、日志、监控、审计、配置等基础设施能力
- 多业务共用且变化频率低的组件 / 工具 / 适配器

不应过早共享的内容：

- 仍在快速变化的业务模型
- 仅在一个核心域里成立的规则
- 看似通用、实际耦合上下文的“万能基类”或“万能 helper”

### 2.5 存量系统适配性判定

对于已有工程，至少做以下五类判断：

| 判断结果 | 适用情形 | 动作 |
|----------|----------|------|
| 直接复用 | 边界与目标模块基本一致 | 保持结构，只做增量扩展 |
| 局部重构 | 整体方向正确，但职责混杂 | 抽取接口、拆职责、补边界 |
| 拆分 / 合并 | 一个模块过大或多个模块职责重叠 | 调整工程组织 |
| 新增独立模块 | 新需求跨域明显或不宜继续塞入旧模块 | 创建独立模块 |
| 替换架构方式 | 现有模式已严重阻碍演进 | 明确迁移与回滚路径 |

### 2.6 Phase 4 输出检查

完成 Phase 4 前，至少应具备：

- 工程模块清单
- 模块职责与依赖说明
- 模块对外接口与数据归属说明
- 存量系统适配性结论

## 3. Phase 5：技术架构与核心选型

### 3.1 先说明系统如何运转

技术架构首先解释运行机制，而不是先列技术名词。

至少说明以下几条链路：

- **请求链路**：入口、鉴权、编排、调用、响应
- **事件链路**：事件发布、消费、失败补偿、重试
- **任务链路**：异步任务、调度、队列、幂等
- **数据链路**：写入、读取、缓存、一致性、回流
- **状态链路**：状态如何变化、由谁拥有、如何恢复

### 3.2 运行链路模板

```text
入口: [用户/API/任务/事件]
鉴权/校验: [方式]
应用编排: [谁负责流程组织]
领域处理: [核心规则在哪一层执行]
基础设施交互: [DB/Cache/MQ/第三方]
失败处理: [重试/补偿/降级/告警]
可观测性: [日志/指标/trace]
```

### 3.3 代码分层设计

可以采用分层架构、六边形架构、模块化单体等形式，但必须明确职责边界。

常见分层参考：

| 层 | 主要职责 | 不应承担的职责 |
|----|----------|----------------|
| 表现层 / 接入层 | 输入输出适配、协议转换、权限入口 | 核心业务规则 |
| 应用层 | 用例编排、事务边界、流程组织 | 基础设施细节 |
| 领域层 | 核心业务规则、领域对象、状态机 | 页面拼装、第三方 SDK 细节 |
| 基础设施层 | DB、缓存、消息、第三方接入 | 业务决策本身 |

### 3.4 复用与内聚设计

技术架构必须同时回答“如何复用”与“如何不被复用拖垮”。

优先遵循：

- 核心业务保持高内聚，优先在模块内部闭环
- 平台能力通过稳定接口被复用，而不是共享内部状态
- 复用优先抽象协议、接口和组件，不优先抽象业务判断
- 为未来扩展保留接缝，但不要为了假想需求提前分布式化

### 3.5 扩展性设计检查

至少回答以下问题：

- 新增一个业务能力，需要改哪些层和哪些模块
- 新增一个页面 / 端侧，需要复用哪些应用或领域能力
- 新增一个第三方集成，需要通过哪类适配层接入
- 如果访问量翻倍，先扩哪一层
- 如果未来需要拆服务，现在应保留哪些接缝

### 3.6 技术选型流程

```text
工程约束确认 -> 候选方案收集 -> 评估矩阵打分 -> 必要时做 POC -> ADR 决策
```

只有在工程架构和运行链路清楚之后，技术选型才有意义。

### 3.7 评估维度建议

| 维度 | 关注点 | 建议权重 |
|------|--------|----------|
| 业务匹配度 | 是否适配当前业务形态和扩展方式 | 30% |
| 工程适配度 | 是否适配当前工程结构、团队协作和代码边界 | 20% |
| 技术成熟度 | 社区、文档、生产案例、维护活跃度 | 20% |
| 运维友好度 | 部署、监控、排障、容量扩展 | 15% |
| 团队成本 | 学习曲线、招聘、迁移、人力成本 | 15% |

### 3.8 常见架构模式选择

| 模式 | 适用场景 | 优点 | 风险 |
|------|----------|------|------|
| 模块化单体 | 业务边界已清楚，但团队规模和复杂度尚可控 | 交付快、认知负担低、易演进 | 模块边界不清时会再次耦合 |
| 微服务 | 边界稳定、团队较大、需独立部署与扩容 | 发布独立、团队自治、弹性更强 | 治理、运维、协同成本高 |
| 事件驱动 | 跨模块异步协作多、强解耦需求高 | 时序解耦、扩展性好 | 一致性、排障和幂等复杂 |

### 3.9 ADR 模板

```markdown
# ADR-[编号]: [决策标题]

## 状态
[提议/接受/废弃/替代]

## 上下文
[描述当前工程结构、业务目标和约束]

## 决策
[做出的技术决策]

## 候选方案
| 方案 | 优势 | 劣势 | 适配结论 |
|------|------|------|----------|
| A | ... | ... | ... |
| B | ... | ... | ... |
| C | ... | ... | ... |

## 后果
[说明收益、代价、风险、回滚点]

## 参考
[POC、文档、基准测试、相关链接]
```

### 3.10 C4 视角建议

为了让工程架构与技术架构可视化，建议至少绘制：

- **L1 System Context**：系统与外部角色/系统关系
- **L2 Container**：主要应用、数据库、MQ、缓存、网关等运行容器
- **L3 Component**：关键容器内部的模块 / 组件边界

如果只是展示运行拓扑而不展示抽象模块，说明图还不够支撑开发。
