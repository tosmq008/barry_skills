# 技术架构设计指南

## 1. 技术选型方法论

### 1.1 选型评估流程

```
需求分析 → 候选方案收集 → 评估矩阵打分 → POC验证 → 最终决策 → ADR记录
```

### 1.2 评估维度详解

**功能匹配度（权重30%）：**
- 是否满足核心功能需求
- 是否支持所需的集成方式
- 是否有必要的扩展点

**技术成熟度（权重20%）：**
- 社区活跃度（GitHub Stars/Issues/PR）
- 版本迭代频率
- 生产环境案例数量
- 文档完善程度

**团队适配度（权重25%）：**
- 团队现有技术栈匹配
- 学习曲线评估
- 招聘市场人才供给

**运维友好度（权重15%）：**
- 部署复杂度
- 监控/日志支持
- 故障排查便利性

**成本评估（权重10%）：**
- 许可证费用
- 基础设施成本
- 运维人力成本

### 1.3 ADR（架构决策记录）模板

```markdown
# ADR-[编号]: [决策标题]

[描述做出的决策]

## 候选方案
| 方案 | 优势 | 劣势 |
|------|------|------|

| 方案B | ... | ... |

## 后果
[描述决策带来的影响，包括正面和负面]

## 参考
[相关文档、基准测试、POC结果链接]
```

## 2. 架构模式详解

### 2.1 单体架构

**适用条件：**
- 团队规模 < 10人
- 业务复杂度低-中
- 快速迭代需求强

**关键设计：**
- 模块化分包（按业务域）
- 清晰的模块间接口
- 数据库按模块分Schema
- 预留拆分接缝

### 2.2 微服务架构

**拆分原则：**
1. 单一职责 - 一个服务一个业务域
2. 数据自治 - 每个服务独立数据库
3. 接口稳定 - API版本化管理
4. 独立部署 - 服务可独立发布

**微服务基础设施：**

| 组件 | 职责 | 候选方案 |
|------|------|----------|
| 服务注册发现 | 服务寻址 | Nacos/Consul/etcd |
| 配置中心 | 配置管理 | Nacos/Apollo/Consul |
| API网关 | 路由/限流/认证 | Kong/APISIX/Spring Cloud Gateway |
| 服务网格 | 流量管理 | Istio/Linkerd |
| 链路追踪 | 调用链追踪 | Jaeger/SkyWalking/Zipkin |
| 日志聚合 | 日志收集分析 | ELK/Loki+Grafana |

### 2.3 事件驱动架构

**事件类型：**
- 领域事件 - 业务状态变化
- 集成事件 - 跨服务通信
- 命令事件 - 触发操作执行

**事件设计规范：**
```json
{
  "event_id": "uuid",
  "event_type": "order.created",
  "source": "order-service",
  "timestamp": "2024-01-01T00:00:00Z",
  "data": {},
  "metadata": {
    "trace_id": "xxx",
    "version": "1.0"
  }
}
```

## 3. C4 模型

> C4 模型提供四个层次的架构视图，从高层到低层逐步细化。

### 3.1 四层视图

| 层级 | 名称 | 受众 | 描述内容 |
|------|------|------|----------|
| L1 | System Context | 所有人 | 系统与外部用户/系统的关系 |
| L2 | Container | 技术人员 | 系统内部的容器（应用、数据库、MQ等） |
| L3 | Component | 开发人员 | 容器内部的组件（模块、服务、控制器等） |
| L4 | Code | 开发人员 | 组件内部的代码结构（类图、ER图等） |

### 3.2 各层绘制要点

**L1 System Context：**
- 中心放置目标系统（一个方框）
- 周围放置用户角色和外部系统
- 用箭头标注交互关系和数据流向
- 不涉及内部技术细节

**L2 Container：**
- 展开目标系统内部
- 标注每个容器的技术选型（如 Vue SPA、FastAPI、PostgreSQL）
- 标注容器间的通信方式（HTTP、gRPC、MQ）
- 标注数据存储类型

**L3 Component：**
- 展开某个容器内部
- 标注主要组件（Controller、Service、Repository）
- 标注组件间的依赖关系
- 标注外部接口

**L4 Code：**
- 仅在需要时绘制
- 类图、ER图、状态图等
- 通常由开发人员在实现阶段补充

## 4. 分层架构设计

### 4.1 层间依赖规则

- 上层可以依赖下层
- 下层不能依赖上层
- 同层之间通过接口通信
- 跨层调用需要通过中间层

### 4.2 各层职责

| 层 | 职责 | 技术关注点 |
|----|------|------------|
| 接入层 | 流量入口，安全防护 | 负载均衡、WAF、限流 |
| 应用层 | 请求编排，数据聚合 | BFF、API组合、缓存 |
| 领域层 | 核心业务逻辑 | 领域模型、业务规则 |
| 基础设施层 | 技术实现细节 | 数据库、缓存、MQ |

## 5. 输出物清单

- 技术选型报告
- 架构决策记录（ADR）
- 系统架构图（C4模型 L1-L3）
- 分层架构设计文档
- 中间件选型方案
- 通信机制设计文档
