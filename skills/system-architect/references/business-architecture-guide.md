# 业务架构与功能架构设计指南

> 本指南服务于 `SKILL.md` 的 **Phase 2（业务功能设计与关联分析）** 和 **Phase 3（功能架构与模块抽象映射）**。
> 目标不是输出一套抽象的 DDD 术语，而是从 PRD 和系统现状中还原业务行为、功能关系和模块边界，为后续工程架构设计提供稳定输入。

## 1. 使用边界

### Phase 2 要解决的问题

- 系统到底要支持哪些业务功能，而不只是页面或接口列表
- 角色、场景、流程、状态、规则之间如何关联
- 哪些功能必须一起设计，哪些应该解耦
- 哪些功能是核心域，哪些只是支撑或集成能力

### Phase 3 要解决的问题

- 如何从 `L1-feature-architecture` 抽象出稳定模块
- 功能节点与模块边界如何映射
- 模块间如何协作、隔离和保护边界
- 哪些抽象可以直接进入工程架构设计

### 本指南的核心输出

- 业务角色与场景矩阵
- 核心业务流程与异常流程清单
- 功能关联矩阵
- 业务规则清单
- 功能架构图
- 功能节点到抽象模块映射表
- 模块边界与协作关系说明

## 2. Phase 2：业务功能设计与关联分析

### 2.1 先还原业务系统，而不是先拆技术模块

在 Phase 2，先回答以下问题：

1. 系统服务哪些角色
2. 每类角色在什么场景下触发什么目标
3. 系统必须完成哪些关键动作
4. 哪些状态会变化
5. 哪些规则决定流程分支、权限和数据约束

如果还停留在“有登录、列表、详情、编辑”这种页面级清单，说明业务功能设计还没完成。

### 2.2 角色-场景-目标矩阵

先把角色和场景讲清楚，再进入流程与能力抽象。

| 角色 | 典型场景 | 用户目标 | 系统职责 | 成功标志 |
|------|----------|----------|----------|----------|
| [角色] | [场景] | [目标] | [必须提供的能力] | [结果] |

### 2.3 业务流程分析模板

对每个核心流程至少输出以下内容：

```text
流程名称: [名称]
流程目标: [目标]
触发角色: [角色]
触发条件: [条件]
前置状态: [状态]

主流程:
  1. [角色/系统] -> [动作] -> [输出]
  2. [角色/系统] -> [动作] -> [输出]

异常流程:
  2a. [异常条件] -> [处理动作] -> [恢复/终止]

后置状态:
  - [状态变化]

关键规则:
  - [规则]

关键数据:
  - [实体/字段]
```

### 2.4 功能关联分析维度

功能之间的关系至少从以下维度分析：

- **触发关系**：A 是否触发 B
- **前后依赖**：A 未完成时 B 能否执行
- **数据共享**：A 与 B 是否共享核心数据对象
- **权限约束**：不同角色是否看到同一功能但执行不同动作
- **状态流转**：一个功能是否依赖另一个功能产生的状态
- **同步 / 异步协作**：链路是即时响应还是事件驱动
- **异常联动**：失败后是否要回滚、补偿、重试、通知

### 2.5 功能关联矩阵模板

| 功能A | 功能B | 关系类型 | 说明 | 设计含义 |
|------|------|----------|------|----------|
| [A] | [B] | [触发/依赖/共享数据/权限/异步] | [说明] | [应同域/应解耦/需事件化] |

### 2.6 业务规则清单模板

| 规则ID | 规则名称 | 触发条件 | 执行动作 | 作用范围 | 优先级 |
|--------|----------|----------|----------|----------|--------|
| BR-001 | [名称] | [条件] | [动作] | [流程/模块] | [高/中/低] |

### 2.7 Phase 2 退出判断

完成 Phase 2 之前，必须能回答：

- 哪些功能构成核心业务闭环
- 哪些功能共享同一业务对象和状态
- 哪些功能必须在一个边界内设计
- 哪些功能更适合通过接口或事件解耦

## 3. Phase 3：功能架构与模块抽象映射

### 3.1 从功能架构图抽象模块，而不是从技术栈倒推

推荐使用以下映射链路：

```text
角色/场景 -> 业务流程 -> 功能节点 -> 能力分组 -> 抽象模块 -> 模块边界
```

抽象模块时，优先看以下信号：

- 是否围绕同一业务目标持续协作
- 是否围绕同一核心实体或状态演进
- 是否有稳定的一组规则和行为
- 是否需要独立演进、独立扩展或独立隔离

### 3.2 模块分类建议

将抽象模块至少分为以下四类：

| 模块类型 | 定义 | 典型特征 |
|----------|------|----------|
| 核心业务模块 | 直接承载核心价值 | 规则复杂、状态关键、演进频繁 |
| 支撑模块 | 服务核心流程但不直接定义产品价值 | 权限、通知、审核、配置等 |
| 平台能力模块 | 被多个业务模块复用的稳定能力 | 身份、文件、搜索、工作流、规则引擎 |
| 外部集成模块 | 对接第三方或遗留系统 | 防腐层、适配层、发布语言 |

### 3.3 功能节点到抽象模块映射表

| 功能节点 | 所属流程 | 抽象模块 | 核心对象/数据 | 协作接口 | 边界说明 |
|----------|----------|----------|----------------|----------|----------|
| [功能] | [流程] | [模块] | [对象] | [接口/事件] | [为何归入此模块] |

### 3.4 模块边界判断原则

- 同一事务一致性强依赖的能力优先放在同一模块
- 同一业务语言、规则和状态机优先放在同一模块
- 只是“前后调用”不代表必须同模块
- 共享数据不等于共享数据库或共享内部实现
- 同一能力被多个业务重复使用时，先判断它是平台能力还是被过度抽象

### 3.5 上下文映射与边界保护

当模块之间存在协作关系时，明确采用哪种边界保护方式：

| 方式 | 适用场景 | 作用 |
|------|----------|------|
| 共享内核 | 强协作且共享稳定模型 | 降低重复建模成本 |
| 客户-供应商 | 明确上下游依赖 | 稳定接口约束 |
| 防腐层 | 对接第三方或遗留系统 | 隔离外部污染 |
| 开放主机服务 | 一个模块服务多个消费方 | 对外标准化输出 |
| 发布语言 | 跨模块交换公共数据格式 | 降低耦合 |
| 领域事件 | 异步协作、最终一致性 | 解耦时序和职责 |

### 3.6 既有工程场景下的映射校验

对于存量系统，必须把“目标模块”与“现有模块”做一轮对照：

| 目标模块 | 现有承接点 | 现状判断 | 处理策略 |
|----------|------------|----------|----------|
| [目标模块] | [目录/服务/包] | [匹配/部分匹配/错位] | [复用/重构/拆分/新增] |

如果一个现有模块同时覆盖多个不稳定业务边界，或一个核心业务闭环被散落在多个不相关模块中，就要在工程架构阶段明确提出调整。

### 3.7 Phase 3 退出判断

完成 Phase 3 之前，必须具备：

- 一张功能架构图
- 一份功能节点到抽象模块映射表
- 一份模块协作与边界说明
- 对现有工程的初步适配性判断

## 4. DDD 作为辅助方法，而不是目标本身

DDD 在本 skill 中主要用来帮助做边界判断，不是为了输出术语。

### 4.1 限界上下文划分原则

1. 语言边界：同一术语在不同语境含义不同，优先拆分
2. 业务边界：流程目标不同，优先拆分
3. 状态边界：状态机不同，优先拆分
4. 团队边界：长期由不同团队负责时，可作为拆分信号
5. 数据边界：核心数据模型差异大时，可辅助拆分

### 4.2 常见对象类型

| 对象类型 | 用途 |
|----------|------|
| 实体 | 具备标识且会演化的业务对象 |
| 值对象 | 无标识、强调属性组合 |
| 聚合根 | 事务一致性与对外访问边界 |
| 领域服务 | 不适合挂在单一实体上的业务规则 |
| 领域事件 | 跨模块或跨聚合协作 |
| 仓储 | 聚合持久化接口 |
