# S3: 新系统设计执行手册

> 重在"全面"——从零开始设计完整系统架构，所有 Phase 无裁剪执行。

## 适用场景

- 全新项目从零开始
- 独立子系统的完整设计
- 产品重构（完全重写）

## 输入与输出

| 项目 | 内容 |
|------|------|
| **输入** | PRD / 产品方案（product-expert 输出） |
| **输出** | `docs/architecture/technical-architecture-design.md` |
| **交付对象** | tech-manager（流转） |

## 执行步骤

### Step 1: 完整需求分析

- 读取 product-expert 输出的全套 PRD 文档
- 功能需求分解（功能分解矩阵）
- 非功能需求量化（NFR 评估矩阵）
- 系统边界与约束识别

### Step 2: 业务架构设计

- DDD 战略设计：核心域/支撑域/通用域
- 限界上下文映射
- 核心业务流程梳理
- 领域模型设计

### Step 3: 技术架构设计

- 技术选型评估（评估矩阵 + web_search 辅助）
- 架构模式选择
- 分层架构设计
- ADR 记录

### Step 4-5: 后端 + 前端架构设计（可并行）

- 后端：服务拆分、API设计、数据库选型、缓存/MQ策略
- 前端：框架选型、工程结构、状态管理、性能目标

### Step 6: 前后端综合架构

- 契约驱动开发流程
- 认证鉴权方案
- 数据交互规范
- 错误码体系

### Step 7-9: 数据 + 安全 + 运维架构（可并行）

- 数据：建模、存储选型、数据流、一致性
- 安全：STRIDE分析、数据分级、合规
- 运维：部署架构、CI/CD、监控、高可用

### Step 10: 输出完整方案 → 流转 tech-manager

流转动作（详见 `references/handoff-guide.md`）：
1. 验证交付物完整性（主文档 + API契约 + 数据字典 + ADR）
2. 在流转指令中包含完整架构方案、全量API清单和数据模型
3. 调用 tech-manager skill，传递流转指令
4. tech-manager 从零搭建项目，按实施计划分阶段推进

## S3 特有指引

### 技术选型 POC 验证流程

对于关键技术选型（框架、数据库、中间件），必须进行 POC 验证：

```
1. 定义 POC 目标（验证什么？成功标准是什么？）
2. 搭建最小验证环境
3. 实现核心场景（2-3个关键用例）
4. 收集数据（性能指标、开发体验、问题清单）
5. 输出 POC 报告 → 纳入 ADR 决策依据
```

POC 触发条件（满足任一）：
- 团队从未使用过该技术
- 候选方案评分差距 < 10%，难以决策
- 涉及性能敏感场景（高并发/大数据量）
- 涉及付费/商业许可的技术

### MVP 范围界定方法

| 步骤 | 动作 | 输出 |
|------|------|------|
| 1. 功能分级 | 按 MoSCoW 对功能分级 | Must/Should/Could/Won't |
| 2. MVP 圈定 | MVP = 全部 Must Have + 部分 Should Have | MVP 功能清单 |
| 3. 架构裁剪 | MVP 阶段可简化的架构决策 | 简化方案（如先单体后拆分） |
| 4. 预留接缝 | 标记 MVP 后需要扩展的架构点 | 扩展点清单 |

### 架构评审会议组织建议

```
评审时机: Phase 3 技术架构设计完成后、Phase 10 最终输出前
参与角色: 架构师(主讲) + 技术负责人 + 前端/后端 Tech Lead + 运维
评审内容:
  - 技术选型合理性（ADR 逐项过）
  - 架构模式匹配度
  - 性能/安全/运维方案可行性
  - 实施计划风险评估
评审输出: 评审意见 → 方案修订 → 定稿
```

## Phase 执行范围

所有 Phase (1-10) **完整执行，无裁剪**。
