# S2: 增量需求落地执行手册

> 重在"适配"——在现有系统架构约束下，设计新需求的落地方案。

## 适用场景

- PRD 已产出，需在现有系统中实现新功能
- 现有功能的重大迭代升级
- 新增业务模块到现有系统

## 输入与输出

| 项目 | 内容 |
|------|------|
| **输入** | PRD + 现有系统代码/文档 |
| **输出** | `docs/architecture/technical-architecture-design.md` |
| **交付对象** | tech-manager（流转） |

## 执行步骤

### Step 1: 需求理解 + 现有系统摸底

- 读取 PRD 文档，提取功能需求和非功能需求
- 扫描现有代码库，理解当前架构和技术栈
- 识别现有系统的架构约束（不可变更的部分）

### Step 2: 影响面分析

核心问题：新需求会影响现有系统的哪些部分？

```
影响面分析维度：
├── 服务层: 哪些服务需要修改？是否需要新增服务？
├── 接口层: 哪些 API 需要修改？需要新增哪些 API？
├── 数据层: 哪些表需要修改？需要新增哪些表？
├── 前端层: 哪些页面/组件需要修改？需要新增哪些？
├── 中间件: 是否需要新增缓存/MQ/ES 等？
└── 配置层: 是否需要新增配置项？
```

**变更影响矩阵（模板）：**

| 影响范围 | 受影响对象 | 变更类型 | 风险等级 | 回归测试重点 |
|----------|------------|----------|----------|--------------|
| 服务层 | [服务名] | 新增/修改/删除 | [高/中/低] | [测试点] |
| 接口层 | [API路径] | 新增/修改 | [高/中/低] | [测试点] |
| 数据层 | [表名] | 新增字段/新表 | [高/中/低] | [测试点] |
| 前端 | [页面/组件] | 新增/修改 | [高/中/低] | [测试点] |

### Step 3: 增量架构设计

- 在现有架构约束下设计技术方案
- 原则：**变更范围最小化**，在满足需求的前提下最小化改动
- 评估是否需要架构调整（如果需要，升级为 S4 场景）

### Step 4: 增量接口契约

- 新增/修改的 API 契约设计
- 确保与现有 API 风格一致（命名、响应格式、错误码）
- 评估对现有 API 消费方的影响（向后兼容性）

### Step 5: 回归风险评估

- 识别变更可能引入的回归问题
- 制定回归测试重点
- 评估是否需要灰度发布

### Step 6: 输出方案 → 流转 tech-manager

流转动作（详见 `references/handoff-guide.md`）：
1. 验证交付物完整性（主文档 + API契约 + 数据字典）
2. 在流转指令中标注增量变更范围和受影响的现有服务/接口
3. 调用 tech-manager skill，传递流转指令
4. tech-manager 需特别关注与现有代码的兼容性和回归风险

## Phase 执行范围

| Phase | S2 执行模式 |
|-------|-------------|
| Phase 1 | 需求分析 + 现有系统影响面分析 |
| Phase 2 | 增量业务域分析（哪些域受影响） |
| Phase 3 | 在现有架构约束下设计技术方案 |
| Phase 4 | 增量变更设计（改哪些服务、加哪些接口） |
| Phase 5 | 增量变更设计（改哪些页面、加哪些组件） |
| Phase 6 | 增量接口契约设计 |
| Phase 7 | ○按需（涉及数据模型变更时执行） |
| Phase 8 | ○按需（涉及安全相关变更时执行） |
| Phase 9 | ○按需（涉及新服务部署时执行） |

> ○按需 判断标准见 SKILL.md 场景路由表。
