# S2: 增量需求落地执行手册

> 重在“适配”——在现有系统架构约束下，为新需求设计可落地的增量方案。

## 适用场景

- PRD 已产出，需在现有系统中实现新功能
- 现有功能的重大迭代升级
- 新增业务模块到现有系统

## 输入与输出

| 项目 | 内容 |
|------|------|
| **输入** | PRD + 现有系统代码/文档 |
| **输出** | `docs/architecture/technical-architecture-design.md` |
| **交付对象** | tech-manager（流转） |

## 执行步骤

### Step 1: Phase 1 需求理解 + 现有系统摸底

- 读取 PRD 文档，提取功能需求、非功能需求和约束
- 扫描现有代码库，理解当前架构和技术栈
- 识别现有系统的架构约束与复用资产
- 明确变更目标、现状边界和成功标准

### Step 2: Phase 2 增量业务功能与关联分析

核心问题：新需求会如何改变现有业务闭环？

- 哪些角色、场景、流程会新增或变化
- 哪些功能之间新增了依赖、权限或状态关系
- 哪些旧功能会被影响或重用

**变更影响矩阵（模板）：**

| 影响范围 | 受影响对象 | 变更类型 | 风险等级 | 回归测试重点 |
|----------|------------|----------|----------|--------------|
| 服务层 | [服务名] | 新增/修改/删除 | [高/中/低] | [测试点] |
| 接口层 | [API路径] | 新增/修改 | [高/中/低] | [测试点] |
| 数据层 | [表名] | 新增字段/新表 | [高/中/低] | [测试点] |
| 前端 | [页面/组件] | 新增/修改 | [高/中/低] | [测试点] |

### Step 3: Phase 3 增量功能架构与模块映射

- 识别新增功能应归入哪个抽象模块
- 判断是否复用现有模块、扩展现有模块或新增模块
- 明确功能节点到目标模块的映射关系
- 识别跨模块协作、边界保护和回归风险

### Step 4: Phase 4 工程架构适配性判断

在现有工程结构下做出明确结论：

- 可直接复用
- 需要局部重构
- 需要拆分 / 合并
- 需要新增独立模块
- 需要升级为 S4（重构迁移）

原则：**在满足需求的前提下最小化改动，但不能为了最小改动牺牲长期边界。**

### Step 5: Phase 5 增量技术架构与核心决策

- 在已确认的工程边界上设计运行链路和代码分层
- 明确新增链路对鉴权、缓存、消息、部署的影响
- 记录技术决策与 ADR
- **遵循 `SKILL.md` 的 Rule 4：Phase 5 完成后先输出摘要，确认后再进入详细设计**

### Step 6: Phase 6-11 增量详细设计

- **Phase 6**：后端增量变更设计（改哪些服务、加哪些接口）
- **Phase 7**：前端增量变更设计（改哪些页面、加哪些组件）
- **Phase 8**：集成与接口契约设计（新增/修改 API、错误码、认证链路）
- **Phase 9**：数据变更设计（涉及表、字段、索引、迁移时执行）
- **Phase 10**：安全变更设计（涉及权限、敏感数据、风控时执行）
- **Phase 11**：运维变更设计（涉及新服务部署、灰度、监控时执行）

### Step 7: Phase 12 输出方案 → 流转 tech-manager

流转动作（详见 `references/handoff-guide.md`）：
1. 验证交付物完整性（主文档 + 系统分析 + 功能架构 + 工程架构 + API契约 + 数据字典）
2. 在流转指令中标注增量变更范围、受影响模块和兼容性风险
3. 调用 tech-manager skill，传递流转指令
4. tech-manager 需特别关注与现有代码的兼容性和回归风险

## Phase 执行范围

| Phase | S2 执行模式 |
|-------|-------------|
| Phase 1 | 需求分析 + 现有系统摸底 + 约束识别 |
| Phase 2 | 增量业务功能设计与功能关联分析 |
| Phase 3 | 增量功能架构与抽象模块映射 |
| Phase 4 | 工程架构适配性判断与变更边界设计 |
| Phase 5 | 增量技术架构与核心 ADR |
| Phase 6 | 增量后端设计 |
| Phase 7 | 增量前端设计 |
| Phase 8 | 增量集成与接口契约设计 |
| Phase 9 | ○按需（涉及数据模型变更时执行） |
| Phase 10 | ○按需（涉及安全相关变更时执行） |
| Phase 11 | ○按需（涉及新服务部署、监控、灰度时执行） |
| Phase 12 | 输出方案并流转 tech-manager |

> 如果 Phase 4 判断现有工程结构已明显不适配当前需求，应升级为 S4 场景。
