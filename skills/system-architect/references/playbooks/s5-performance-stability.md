# S5: 性能/稳定性专项执行手册

> 重在“定点”——针对性能瓶颈或稳定性问题，设计精准优化方案。

## 适用场景

- 系统响应变慢，用户体验下降
- 大促/活动前的性能保障
- SLA 不达标需要优化
- 频繁出现稳定性问题（OOM、超时、雪崩）
- 容量规划（业务增长前的扩容评估）

## 输入与输出

| 项目 | 内容 |
|------|------|
| **输入** | 性能数据 + 监控告警 + 现有架构 |
| **输出** | `docs/architecture/performance-optimization-plan.md` |
| **交付对象** | tech-manager（流转） |

## 执行步骤

### Step 1: Phase 1 瓶颈识别与系统分析

数据收集：
- APM 数据：接口响应时间分布、慢接口 Top N
- 数据库：慢查询日志、连接池使用率、锁等待
- 缓存：命中率、内存使用、热 Key
- 系统资源：CPU/内存/磁盘 IO/网络
- 前端：Core Web Vitals、资源加载瀑布图

瓶颈分类：

```text
瓶颈类型          典型表现                    定位方法
──────────────────────────────────────────────────────────
计算瓶颈          CPU 持续高位               Profiling/火焰图
IO 瓶颈           磁盘IO高/网络延迟          iostat/网络抓包
内存瓶颈          OOM/频繁GC                 内存分析/GC日志
数据库瓶颈        慢查询/连接池满            慢查询日志/explain
缓存瓶颈          缓存穿透/击穿/雪崩        命中率监控
并发瓶颈          线程池满/锁竞争            线程dump/锁分析
前端瓶颈          FCP/LCP超标               Lighthouse/DevTools
```

### Step 2: Phase 2 受影响业务功能与关键链路分析

- 识别性能问题影响了哪些业务闭环和用户场景
- 明确哪些链路是核心路径、哪些是次要路径
- 判断性能问题是否源于业务规则、流程设计或调用时序

### Step 3: Phase 3 受影响功能架构与热点模块分析

- 明确热点功能属于哪些抽象模块
- 判断瓶颈集中在核心业务模块、平台能力还是外部集成
- 识别哪些跨模块协作造成额外延迟或不稳定性

### Step 4: Phase 4 工程边界与优化承接面分析

- 判断瓶颈根因是否来自工程结构、依赖方向、共享能力或发布边界
- 明确优化是在现有工程结构内完成，还是需要做结构性调整
- 输出可作为实施单元的优化边界

### Step 5: Phase 5 技术架构优化方案与核心决策

- 设计技术架构层面的优化路径：缓存、异步化、并发模型、扩缩容、降级限流等
- 明确运行链路、数据链路、失败恢复和可观测性改造点
- 记录优化 ADR
- **遵循 `SKILL.md` 的 Rule 4：Phase 5 完成后暂停，确认后再进入详细优化设计**

### Step 6: Phase 6-11 分维度优化设计

- **Phase 6**：后端性能优化（缓存、并发、数据库、连接池）
- **Phase 7**：前端性能优化（加载、渲染、资源、缓存）
- **Phase 8**：前后端交互优化（批量接口、契约调整、错误处理、重试策略）
- **Phase 9**：数据层优化（索引、查询、冷热分层、分库分表、缓存同步）
- **Phase 10**：安全兜底（确保性能优化不引入权限和数据泄露风险）
- **Phase 11**：运维层优化（监控增强、扩缩容策略、资源调优、压测机制）

### Step 7: Phase 12 输出优化方案 → 流转 tech-manager

流转动作（详见 `references/handoff-guide.md`）：
1. 验证交付物完整性（优化方案 + 压测验证计划 + SLA 目标 + 核心 ADR）
2. 在流转指令中标注优化分级和实施顺序
3. 调用 tech-manager skill，传递流转指令
4. tech-manager 按优先级分批实施，每批完成后需压测验证

## 优化方案分级

| 级别 | 定义 | 示例 | 实施周期 |
|------|------|------|----------|
| Quick Win | 低成本高收益 | 加索引、调连接池、开缓存 | 1-3天 |
| 中期优化 | 需要一定开发量 | 异步化、读写分离、CDN | 1-2周 |
| 长期架构调整 | 架构级变更 | 分库分表、事件化、模块拆分 | 1-3月 |

## 压测验证计划

```text
压测流程：
1. 基准测试（优化前）→ 记录基线数据
2. 实施 Quick Win 优化
3. 回归压测 → 对比基线
4. 实施中期优化
5. 回归压测 → 对比基线
6. 评估是否需要长期架构调整
```

## Phase 执行范围

| Phase | S5 执行模式 |
|-------|-------------|
| Phase 1 | 瓶颈识别与根因定位（Profiling/APM/慢查询） |
| Phase 2 | 受影响业务功能与关键链路分析 |
| Phase 3 | 受影响功能架构与热点模块分析 |
| Phase 4 | 工程边界与优化承接面分析 |
| Phase 5 | 技术架构优化方案与 ADR |
| Phase 6 | 后端性能优化：缓存、并发、数据库、连接池 |
| Phase 7 | 前端性能优化：加载、渲染、资源、缓存 |
| Phase 8 | ○按需（当前后端交互是性能瓶颈时执行） |
| Phase 9 | 数据层优化：索引、查询、分库分表、缓存同步 |
| Phase 10 | ○按需（确保性能优化不引入安全漏洞） |
| Phase 11 | 运维层优化：监控增强、扩缩容策略、资源调优 |
| Phase 12 | 输出优化方案并流转 tech-manager |
