# 场景组合指南

复杂项目可能需要组合多个场景顺序执行。

## 常见组合模式

| 组合模式 | 执行顺序 | 适用场景 |
|----------|----------|----------|
| S1 → S4 | 先评审，再重构 | 接手遗留系统后的架构升级 |
| S1 → S5 | 先评审，再优化 | 系统性能问题的系统性解决 |
| S1 → S2 | 先评审，再落地 | 不熟悉的系统上开发新功能 |
| S3 → S2 | 先新建，再迭代 | 新系统上线后的功能迭代 |
| S5 → S4 | 先优化，再重构 | 性能优化发现需要架构变更 |

## 组合执行原则

1. **独立输出** - 每个场景独立输出文档
2. **引用传递** - 后一个场景可引用前一个场景的输出
3. **一致性** - 场景间的交付物保持一致性
4. **末端流转** - 仅最终场景执行流转至 tech-manager（中间场景输出文档供下一场景引用）

## 流转时机

- 组合场景中，中间场景完成后不流转 tech-manager
- 仅最终场景完成后，汇总所有场景的交付物，统一流转
- 流转指令中需注明组合场景的完整上下文
