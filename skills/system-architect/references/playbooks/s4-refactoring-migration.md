# S4: 架构重构/迁移执行手册

> 重在“过渡”——设计从现有架构到目标架构的安全迁移路径。

## 适用场景

- 单体架构拆分为微服务
- 技术栈升级（如 Python 2→3, Vue 2→3, Java 8→17）
- 框架迁移（如 Django→FastAPI, Express→NestJS）
- 数据库迁移（如 MySQL→PostgreSQL）
- 部署方式升级（如物理机→容器化→K8s）
- 多系统合并或拆分

## 输入与输出

| 项目 | 内容 |
|------|------|
| **输入** | 现有系统 + 目标架构愿景 |
| **输出** | `docs/architecture/migration-plan.md` + `technical-architecture-design.md` |
| **交付对象** | tech-manager（流转） |

## 执行步骤

### Step 1: Phase 1 现状系统分析 + 迁移动机定义

- 复用 S1 的评估方法还原现状架构
- 盘点技术债务、痛点和历史约束
- 明确迁移动机、预期收益和成功标准

### Step 2: Phase 2 目标业务功能与范围界定

- 明确迁移后的业务边界、角色和关键闭环
- 识别哪些能力保持不变，哪些能力借迁移做重构
- 划清“本次迁移范围”和“后续演进范围”

### Step 3: Phase 3 目标功能架构与模块抽象

- 设计迁移后的目标功能架构
- 将旧系统模块映射到目标抽象模块
- 明确哪些旧边界要保留、拆分、合并或下沉

### Step 4: Phase 4 目标工程架构与迁移单元拆分

- 把目标模块落到服务、应用、包、目录和共享库
- 规划迁移单元边界，确保可以分批实施与回滚
- 明确新旧系统在迁移期的承接关系

### Step 5: Phase 5 目标技术架构与迁移 ADR

- 设计目标运行链路、代码分层、基础设施方案
- 选择迁移策略（如 Strangler Fig / 并行运行 / 大爆炸 / 分支抽象）
- 记录迁移 ADR、风险与回滚点
- **遵循 `SKILL.md` 的 Rule 4：Phase 5 完成后暂停，确认后再进入详细迁移设计**

### Step 6: Phase 6-11 分维度迁移设计

- **Phase 6**：目标后端架构 + 服务迁移方案
- **Phase 7**：目标前端架构 + 渐进迁移方案
- **Phase 8**：新旧系统并行期的接口兼容与集成方案
- **Phase 9**：数据迁移方案：兼容性、迁移脚本、回滚
- **Phase 10**：迁移期安全方案：权限、密钥、数据保护不降级
- **Phase 11**：迁移期运维方案：新旧并行部署、流量切换、回滚机制

### Step 7: Phase 12 输出迁移方案 → 流转 tech-manager

流转动作（详见 `references/handoff-guide.md`）：
1. 验证交付物完整性（迁移方案 + 系统分析 + 功能架构 + 工程架构 + 目标技术架构 + API契约 + 数据字典）
2. 在流转指令中标注迁移策略和灰度切换策略
3. 调用 tech-manager skill，传递流转指令
4. tech-manager 需同时维护新旧系统，按迁移计划分批实施

## 差距分析模板

```text
维度          当前状态(As-Is)    目标状态(To-Be)    差距    迁移难度
────────────────────────────────────────────────────────────────────
架构模式      单体               微服务             大      高
技术栈        Django 2.x         FastAPI            中      中
数据库        MySQL 5.7          PostgreSQL 15      中      高
部署方式      物理机             K8s                大      高
监控体系      基础日志           全链路追踪         大      中
```

## 架构演进策略

### 演进原则

1. **渐进式迁移** - 不做大爆炸式重写，逐步替换
2. **Strangler Fig 模式** - 新功能用新架构，旧功能逐步迁移
3. **数据先行** - 先建立数据兼容与同步能力，再迁应用
4. **可回滚** - 每一步都可回滚到上一个稳定状态

### 演进路径模板

| 阶段 | 策略 | 关键动作 | 验证标准 |
|------|------|----------|----------|
| 评估期 | 现状分析 | 技术债务盘点、架构评估 | 评估报告 |
| 准备期 | 基础设施 | 新架构基础搭建、CI/CD、数据同步 | 新环境就绪 |
| 迁移期 | Strangler Fig | 新功能新架构、旧功能逐步迁移 | 功能对等 |
| 切换期 | 流量切换 | 灰度切换、旧系统下线 | 全量切换 |
| 优化期 | 清理优化 | 旧代码清理、性能优化 | 技术债务收敛 |
