# S4: 架构重构/迁移执行手册

> 重在"过渡"——设计从现有架构到目标架构的安全迁移路径。

## 适用场景

- 单体架构拆分为微服务
- 技术栈升级（如 Python 2→3, Vue 2→3, Java 8→17）
- 框架迁移（如 Django→FastAPI, Express→NestJS）
- 数据库迁移（如 MySQL→PostgreSQL）
- 部署方式升级（如 物理机→容器化→K8s）
- 多系统合并或拆分

## 输入与输出

| 项目 | 内容 |
|------|------|
| **输入** | 现有系统 + 目标架构愿景 |
| **输出** | `docs/architecture/migration-plan.md` + `technical-architecture-design.md` |
| **交付对象** | tech-manager（流转） |

## 执行步骤

### Step 1: 现状分析 + 目标定义

现状分析（复用 S1 的评估方法）：
- 当前架构还原
- 技术债务盘点
- 痛点识别

目标定义：
- 目标架构愿景（清晰描述迁移后的理想状态）
- 迁移动机和预期收益
- 成功标准定义

### Step 2: 差距分析 (Gap Analysis)

```
维度          当前状态(As-Is)    目标状态(To-Be)    差距    迁移难度
────────────────────────────────────────────────────────────────────
架构模式      单体               微服务             大      高
技术栈        Django 2.x         FastAPI            中      中
数据库        MySQL 5.7          PostgreSQL 15      中      高
部署方式      物理机             K8s                大      高
监控体系      基础日志           全链路追踪         大      中
```

### Step 3: 迁移策略选择

| 策略 | 适用场景 | 风险 | 周期 |
|------|----------|------|------|
| Strangler Fig | 大型系统渐进迁移 | 低 | 长 |
| 并行运行 | 关键系统零停机迁移 | 中 | 中 |
| 大爆炸 | 小型系统/紧急迁移 | 高 | 短 |
| 分支抽象 | 内部模块替换 | 低 | 中 |

### Step 4: 迁移路径规划

```
阶段1: 准备期
├── 新架构基础搭建（脚手架、CI/CD、基础服务）
├── 数据同步机制建立（双写/CDC）
└── 流量路由机制准备（API Gateway/Feature Flag）

阶段2: 迁移期
├── 新功能直接在新架构开发
├── 旧功能按优先级逐步迁移
├── 每个迁移单元独立验证
└── 保持新旧系统数据一致

阶段3: 切换期
├── 灰度切换流量到新系统
├── 监控对比新旧系统指标
├── 确认无问题后全量切换
└── 保留旧系统回滚能力

阶段4: 清理期
├── 旧系统下线
├── 数据同步机制清理
├── 兼容代码清理
└── 文档更新
```

### Step 5: 数据迁移方案

关键考虑：
- 数据格式兼容性（Schema 变更映射）
- 迁移脚本编写与测试
- 增量数据同步（CDC / 双写）
- 数据校验（迁移前后一致性验证）
- 回滚方案（数据可回退）

### Step 6: 输出迁移方案 → 流转 tech-manager

流转动作（详见 `references/handoff-guide.md`）：
1. 验证交付物完整性（迁移方案 + 目标架构文档 + API契约 + 数据字典）
2. 在流转指令中标注迁移策略（Strangler Fig/并行/大爆炸）和灰度切换策略
3. 调用 tech-manager skill，传递流转指令
4. tech-manager 需同时维护新旧系统，按迁移计划分批实施

## Phase 执行范围

| Phase | S4 执行模式 |
|-------|-------------|
| Phase 1 | 现状分析 + 目标架构定义 + 差距分析(Gap Analysis) |
| Phase 2 | 业务域重新划分（如需） |
| Phase 3 | 目标架构设计 + 迁移路径规划 |
| Phase 4 | 目标后端架构 + 服务迁移方案 |
| Phase 5 | 目标前端架构 + 渐进迁移方案 |
| Phase 6 | 新旧系统并行期的接口兼容方案 |
| Phase 7 | 数据迁移方案：兼容性、迁移脚本、回滚 |
| Phase 8 | ○按需（确保迁移过程中安全性不降级） |
| Phase 9 | 迁移期运维方案：新旧并行部署、流量切换、回滚机制 |

## 架构演进策略

### 演进原则

1. **渐进式迁移** - 不做大爆炸式重写，逐步替换
2. **Strangler Fig 模式** - 新功能用新架构，旧功能逐步迁移
3. **数据先行** - 先迁移数据层，再迁移应用层
4. **可回滚** - 每一步都可回滚到上一个稳定状态

### 演进路径模板

| 阶段 | 策略 | 关键动作 | 验证标准 |
|------|------|----------|----------|
| 评估期 | 现状分析 | 技术债务盘点、架构评估 | 评估报告 |
| 准备期 | 基础设施 | 新架构基础搭建、CI/CD | 新环境就绪 |
| 迁移期 | Strangler Fig | 新功能新架构、旧功能逐步迁移 | 功能对等 |
| 切换期 | 流量切换 | 灰度切换、旧系统下线 | 全量切换 |
| 优化期 | 清理优化 | 旧代码清理、性能优化 | 技术债务清零 |
