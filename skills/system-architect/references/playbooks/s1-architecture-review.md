# S1: 架构评审执行手册

> 重在"诊断"——评估现有系统架构健康度，输出评审报告与调整建议。

## 适用场景

- 定期架构健康度检查
- 接手新项目前的架构摸底
- 技术债务评估
- 架构合规性审计

## 输入与输出

| 项目 | 内容 |
|------|------|
| **输入** | 现有代码库 + 系统文档 + 监控数据 |
| **输出** | `docs/architecture/architecture-review-report.md` |
| **交付对象** | 决策层 / 技术负责人（不直接流转 tech-manager） |

## 执行步骤

### Step 1: 信息收集

```bash
# 1. 扫描项目结构
find . -type f -name "*.py" -o -name "*.js" -o -name "*.ts" -o -name "*.go" -o -name "*.java" | head -100
tree -L 3 -I "node_modules|__pycache__|.git|dist|build"

# 2. 分析依赖
cat requirements.txt / package.json / go.mod / pom.xml

# 3. 查看配置
cat docker-compose.yml / Dockerfile / k8s/*.yaml

# 4. 查看现有文档
ls docs/ README.md
```

### Step 2: 架构还原

- 从代码反推实际架构（模块依赖、调用链、数据流）
- 与现有文档对比，识别文档与实际的偏差
- 绘制实际架构图（C4 Container 级别）

### Step 3: 各维度评估

对每个维度进行 1-5 分评估：

| 分数 | 含义 | 判断标准 |
|------|------|----------|
| 1 | 严重问题 | 存在阻塞性问题，需立即修复 |
| 2 | 较多问题 | 多处不合理，需要重点改进 |
| 3 | 基本合格 | 能工作但有改进空间 |
| 4 | 良好 | 设计合理，少量优化点 |
| 5 | 优秀 | 业界最佳实践水平 |

**评估检查清单：**

业务架构评估：
- [ ] 业务域划分是否清晰？是否存在职责混乱？
- [ ] 限界上下文边界是否合理？
- [ ] 核心域是否得到足够关注？

技术架构评估：
- [ ] 分层是否清晰？是否存在跨层调用？
- [ ] 技术选型是否合理？是否存在过时技术？
- [ ] 架构模式是否匹配业务规模？

后端架构评估：
- [ ] 服务拆分粒度是否合理？
- [ ] API 设计是否规范？
- [ ] 数据库设计是否合理？索引是否充分？
- [ ] 缓存策略是否有效？
- [ ] 错误处理是否完善？

前端架构评估：
- [ ] 工程化水平如何？构建/部署是否自动化？
- [ ] 组件复用率如何？
- [ ] 状态管理是否合理？
- [ ] 性能指标（FCP/LCP/CLS）是否达标？

数据架构评估：
- [ ] 数据模型是否规范？
- [ ] 数据一致性保障是否充分？
- [ ] 数据备份和恢复策略是否完善？

安全架构评估：
- [ ] 认证鉴权是否安全？
- [ ] 数据加密/脱敏是否到位？
- [ ] 是否存在已知安全漏洞？

运维架构评估：
- [ ] CI/CD 流水线是否完善？
- [ ] 监控覆盖率如何？
- [ ] 容灾和高可用方案是否充分？
- [ ] 日志是否规范？

### Step 5: 交付与流转判断

- 交付评审报告给决策层/技术负责人
- **不直接流转 tech-manager**
- 根据评审结论判断后续动作：

```
S1 评审完成
    ↓
输出: docs/architecture/architecture-review-report.md
    ↓
交付: 决策层 / 技术负责人
    ↓
决策分支:
├── 无需调整 → 归档报告，结束
├── 需要增量优化 → 启动 S2（增量需求落地）→ 流转 tech-manager
├── 需要架构重构 → 启动 S4（架构重构/迁移）→ 流转 tech-manager
└── 需要性能优化 → 启动 S5（性能/稳定性专项）→ 流转 tech-manager
```

> S1 的评审报告可作为后续 S2/S4/S5 的输入参考文档。

## Phase 执行范围

所有 Phase (1-9) 均以**评估模式**执行，不做设计，只做诊断打分。

| Phase | S1 执行模式 |
|-------|-------------|
| Phase 1 | 系统现状调研（代码扫描、架构还原、技术债务盘点） |
| Phase 2 | 评估现有业务域划分合理性 |
| Phase 3 | 评估现有技术架构合理性 |
| Phase 4 | 评估服务拆分、API规范性、数据库设计质量 |
| Phase 5 | 评估前端工程化水平、组件复用率、构建性能 |
| Phase 6 | 评估前后端集成质量：契约一致性、错误处理 |
| Phase 7 | 评估数据模型质量、存储选型、数据一致性 |
| Phase 8 | 安全漏洞扫描、合规性检查、权限模型评估 |
| Phase 9 | 评估运维成熟度：CI/CD、监控覆盖、容灾能力 |
