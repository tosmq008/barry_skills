# S1: 架构评审执行手册

> 重在“诊断”——评估现有系统架构健康度，输出评审报告与调整建议。

## 适用场景

- 定期架构健康度检查
- 接手新项目前的架构摸底
- 技术债务评估
- 架构合规性审计

## 输入与输出

| 项目 | 内容 |
|------|------|
| **输入** | 现有代码库 + 系统文档 + 监控数据 |
| **输出** | `docs/architecture/architecture-review-report.md` |
| **交付对象** | 决策层 / 技术负责人（不直接流转 tech-manager） |

## 执行步骤

### Step 1: Phase 1 系统现状调研

```bash
# 1. 扫描项目结构
tree -L 3 -I "node_modules|__pycache__|.git|dist|build"

# 2. 分析依赖
cat requirements.txt / package.json / go.mod / pom.xml

# 3. 查看部署与配置
cat docker-compose.yml / Dockerfile / k8s/*.yaml

# 4. 查看现有文档
ls docs/ README.md
```

- 从代码反推实际系统结构、模块依赖、调用链和数据流
- 与现有文档对比，识别“设计稿”和“真实系统”的差距
- 形成现状与约束摘要

### Step 2: Phase 2-5 架构主干评估

按新主流程评估以下内容：

- **Phase 2**：业务功能设计与关联分析是否清晰
- **Phase 3**：功能架构与模块抽象是否合理
- **Phase 4**：工程架构边界、依赖方向、共享能力是否合理
- **Phase 5**：技术架构、运行链路、代码分层、扩展方式是否合理

建议输出中间评审摘要：

- 主要业务/功能边界问题
- 主要工程边界问题
- 主要技术架构问题
- 是否值得继续深入做 Phase 6-11 细评

### Step 3: Phase 6-11 各维度深评

对每个维度进行 1-5 分评估：

| 分数 | 含义 | 判断标准 |
|------|------|----------|
| 1 | 严重问题 | 存在阻塞性问题，需立即修复 |
| 2 | 较多问题 | 多处不合理，需要重点改进 |
| 3 | 基本合格 | 能工作但有改进空间 |
| 4 | 良好 | 设计合理，少量优化点 |
| 5 | 优秀 | 业界最佳实践水平 |

**评估检查清单：**

业务功能与功能架构评估：
- [ ] 业务闭环是否清晰？功能之间是否存在职责混乱？
- [ ] 功能架构与抽象模块边界是否合理？
- [ ] 核心业务模块是否得到足够关注？

工程架构与技术架构评估：
- [ ] 工程模块边界是否清晰？依赖方向是否稳定？
- [ ] 共享能力是否被合理抽象，是否存在万能 common？
- [ ] 运行链路、代码分层、技术选型是否匹配业务规模？

后端架构评估：
- [ ] 服务拆分粒度是否合理？
- [ ] API 设计是否规范？
- [ ] 数据库设计是否合理？索引是否充分？
- [ ] 缓存策略是否有效？
- [ ] 错误处理是否完善？

前端架构评估：
- [ ] 工程化水平如何？构建/部署是否自动化？
- [ ] 组件复用率如何？
- [ ] 状态管理是否合理？
- [ ] 性能指标（FCP/LCP/CLS）是否达标？

集成/数据/安全/运维评估：
- [ ] 前后端契约是否一致？错误码和认证链路是否统一？
- [ ] 数据模型是否规范？一致性保障是否充分？
- [ ] 认证鉴权、数据保护、依赖安全是否到位？
- [ ] CI/CD、监控、容灾、日志规范是否充分？

### Step 5: 交付与流转判断

- 交付评审报告给决策层/技术负责人
- **不直接流转 tech-manager**
- 根据评审结论判断后续动作：

```text
S1 评审完成
    ↓
输出: docs/architecture/architecture-review-report.md
    ↓
交付: 决策层 / 技术负责人
    ↓
决策分支:
├── 无需调整 → 归档报告，结束
├── 需要增量优化 → 启动 S2（增量需求落地）→ 流转 tech-manager
├── 需要架构重构 → 启动 S4（架构重构/迁移）→ 流转 tech-manager
└── 需要性能优化 → 启动 S5（性能/稳定性专项）→ 流转 tech-manager
```

> S1 的评审报告可作为后续 S2/S4/S5 的输入参考文档。

## Phase 执行范围

所有 Phase 以**评估模式**执行，不做设计，只做诊断和打分。

| Phase | S1 执行模式 |
|-------|-------------|
| Phase 1 | 系统现状调研（代码扫描、架构还原、技术债务盘点） |
| Phase 2 | 评估业务功能设计与关键关联是否清晰 |
| Phase 3 | 评估功能架构与抽象模块边界是否合理 |
| Phase 4 | 评估工程架构：模块边界、依赖方向、共享能力 |
| Phase 5 | 评估技术架构：运行链路、分层方式、技术选型合理性 |
| Phase 6 | 评估后端架构：服务/API/事务/缓存/MQ |
| Phase 7 | 评估前端架构：工程化、组件复用、状态管理、性能 |
| Phase 8 | 评估集成架构：契约一致性、错误处理、认证链路 |
| Phase 9 | 评估数据架构：模型质量、存储选型、数据一致性 |
| Phase 10 | 评估安全架构：漏洞、权限模型、数据保护、依赖安全 |
| Phase 11 | 评估运维架构：CI/CD、监控覆盖、容灾与回滚能力 |
| Phase 12 | 输出评审报告、建议与演进路线图 |
