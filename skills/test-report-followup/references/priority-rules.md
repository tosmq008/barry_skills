# 优先级规则

## 优先级定义

### 问题优先级

| 优先级 | 名称 | 定义 | 处理时限 |
|--------|------|------|----------|
| P0 | 紧急 | 阻塞测试或上线，核心功能不可用 | 立即处理 |
| P1 | 高 | 影响主流程，用户无法完成核心操作 | 当天完成 |
| P2 | 中 | 影响次要功能，有替代方案 | 3天内完成 |
| P3 | 低 | 优化类问题，不影响使用 | 本周内完成 |

### 严重程度与优先级对应

| 严重程度 | 默认优先级 | 说明 |
|----------|------------|------|
| Critical | P0 | 系统崩溃、数据丢失 |
| Major | P1 | 主要功能异常 |
| Minor | P2 | 次要功能异常 |
| Trivial | P3 | 界面/体验问题 |

## 优先级判定规则

### 判定流程

```

│                      优先级判定流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

│         ┌────┴────┐                                            │
│         是        否                                            │
│         │         │                                            │
│         ▼         ▼                                            │
│        P0     继续判断                                          │
│                   │                                            │
│  Step 2: 判断影响范围                                           │
│     └── 是否影响核心流程？                                      │
│              │                                                  │
│         ┌────┴────┐                                            │
│         是        否                                            │
│         │         │                                            │
│         ▼         ▼                                            │
│        P1     继续判断                                          │
│                   │                                            │
│  Step 3: 判断功能重要性                                         │

```

### 判定标准详解

#### P0 - 紧急

**判定条件（满足任一）：**
- 系统无法启动或崩溃
- 核心功能完全不可用
- 数据丢失或损坏
- 安全漏洞
- 阻塞测试进行
- 阻塞版本发布

**示例：**
- 登录功能完全失效
- 支付流程报错无法完成
- 数据库连接失败
- 页面白屏无法加载

#### P1 - 高

**判定条件（满足任一）：**
- 主要功能异常但有临时方案
- 影响大部分用户的操作
- 数据显示错误但不丢失
- 性能严重下降

**示例：**
- 表单提交偶发失败
- 列表数据显示不完整
- 页面加载超过10秒
- 部分用户无法正常使用

#### P2 - 中

**判定条件（满足任一）：**
- 次要功能异常
- 有替代操作方案
- 影响少数用户
- 体验问题但不影响功能

**示例：**
- 排序功能异常
- 筛选条件不准确
- 导出功能失败
- 提示信息不准确

#### P3 - 低

**判定条件（满足任一）：**
- 界面美观问题
- 文案错误
- 优化建议
- 边缘场景问题

**示例：**
- 按钮对齐问题
- 文字颜色不一致
- 错别字
- 极端数据显示问题

## 优先级调整规则

### 可以提升优先级的情况

| 情况 | 调整 | 说明 |
|------|------|------|
| 影响范围扩大 | +1级 | 发现影响更多用户 |
| 发现关联问题 | +1级 | 问题导致其他功能异常 |
| 临近发布 | +1级 | 发布前的问题优先处理 |
| 用户投诉 | +1级 | 收到用户反馈 |

### 可以降低优先级的情况

| 情况 | 调整 | 说明 |
|------|------|------|
| 找到临时方案 | -1级 | 有可行的规避方案 |
| 影响范围缩小 | -1级 | 确认只影响少数场景 |
| 非核心功能 | -1级 | 确认为边缘功能 |

## 处理时限规则

### 各角色响应时限

| 优先级 | 产品经理 | UI设计师 | 前端工程师 | 后端工程师 |
|--------|----------|----------|------------|------------|
| P0 | 2小时 | 4小时 | 立即 | 立即 |
| P1 | 4小时 | 8小时 | 当天 | 当天 |
| P2 | 1天 | 2天 | 3天 | 3天 |
| P3 | 3天 | 1周 | 1周 | 1周 |

### 超时处理规则

```
┌─────────────────────────────────────────────────────────────────┐
│                      超时处理流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 首次超时（超过截止时间）                                    │
│     └── 发送提醒通知                                            │
│                                                                 │
│  2. 二次超时（超过截止时间 50%）                                │
│     └── 升级通知，抄送负责人                                    │
│                                                                 │
│  3. 三次超时（超过截止时间 100%）                               │
│     └── 标记为延期，记录原因                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 问题类型优先级参考

### Bug 类型

| Bug 类型 | 建议优先级 | 说明 |
|----------|------------|------|
| 功能 Bug | P1-P2 | 根据功能重要性判断 |
| UI Bug | P2-P3 | 根据影响程度判断 |
| 交互 Bug | P2 | 影响用户体验 |
| API Bug | P1-P2 | 根据接口重要性判断 |
| 数据 Bug | P0-P1 | 数据问题通常较严重 |
| 性能 Bug | P1-P2 | 根据影响程度判断 |

### 非 Bug 类型

| 问题类型 | 建议优先级 | 说明 |
|----------|------------|------|
| PRD 问题 | P1-P2 | 阻塞开发则 P1 |
| 设计问题 | P2 | 通常不阻塞 |
| 体验问题 | P2-P3 | 优化类问题 |
| 缺失功能 | P2-P3 | 需要评估 |

## 优先级分配示例

### 示例 1: 登录失败

```markdown
问题: 用户登录时提示"系统错误"，无法登录

判定过程:
1. 是否阻塞？ → 是，用户无法使用系统
2. 结论: P0

处理时限: 立即处理
```

### 示例 2: 列表排序异常

```markdown
问题: 用户列表按时间排序时，顺序不正确

判定过程:
1. 是否阻塞？ → 否
2. 是否影响核心流程？ → 否，用户仍可查看列表
3. 是否影响重要功能？ → 是，排序是常用功能
4. 结论: P2

处理时限: 3天内完成
```

### 示例 3: 按钮颜色不一致

```markdown
问题: 确认按钮颜色与设计稿不一致

判定过程:
1. 是否阻塞？ → 否
2. 是否影响核心流程？ → 否
3. 是否影响重要功能？ → 否，仅视觉问题
4. 结论: P3

处理时限: 本周内完成
```

- [ ] 是否有临时的规避方案？
- [ ] 是否与相关人员确认过？
- [ ] 处理时限是否合理？

- [ ] 是否有充分的调整理由？
- [ ] 是否通知了相关人员？
