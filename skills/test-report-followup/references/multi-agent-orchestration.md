# 多 Agent 调度指南

## 概述

测试报告跟进专家作为跟进总监角色，通过调度多个专业 Agent 并行执行修复任务，显著提高问题修复效率。

## Agent 角色详解

### 1. 跟进总监 (Orchestrator)

**职责：**
- 解析测试报告，提取所有问题
- 分解任务并分配给对应 Agent
- 监控各 Agent 执行状态
- 收集汇总修复结果
- 调度测试 Agent 执行回归
- 生成跟进完成报告

**能力要求：**
- 理解测试报告结构
- 任务分解和优先级排序
- 并行调度和依赖管理
- 结果汇总和报告生成

### 2. 产品 Agent

**职责：**
- 处理 PRD 问题和需求澄清
- 评估缺失功能的优先级
- 确认体验问题的改进方向
- 更新 PRD 文档

**输入：**

## 调度策略

### 任务分解原则

1. **按问题类型分组**
   - PRD 问题 → 产品 Agent
   - UI Bug → UI设计 Agent + 前端 Agent
   - 功能 Bug → 前端/后端 Agent
   - API Bug → 后端 Agent

2. **按优先级排序**
   - P0 问题优先处理
   - Critical Bug 立即修复

3. **按依赖关系排序**
   - 无依赖任务并行执行
   - 有依赖任务串行执行

## 错误处理

### Agent 执行失败

| 失败类型 | 处理方式 |
|----------|----------|
| 超时 | 标记为未完成，记录已完成部分 |
| 依赖阻塞 | 等待上游完成或标记为阻塞 |
| 任务失败 | 记录失败原因，继续其他任务 |
| Agent 崩溃 | 重启 Agent，从断点继续 |

### 重试机制

```python
# 重试配置
RETRY_CONFIG = {
    "max_retries": 2,
    "retry_delay": 60,  # 秒
    "retry_on": ["timeout", "connection_error", "agent_crash"]
}
```

### 阻塞处理

```markdown
## 阻塞任务处理流程

1. 识别阻塞任务
   └── 检查任务依赖是否满足

2. 分析阻塞原因
   ├── 上游任务未完成
   ├── 上游任务失败
   └── 资源冲突

3. 处理策略
   ├── 等待上游完成
   ├── 跳过阻塞任务
   └── 人工介入处理

4. 更新任务状态
   └── 标记为 Blocked，记录原因
```

## 最佳实践

### 1. 合理分配任务

- 根据问题数量均衡分配
- 避免单个 Agent 负载过重
- 考虑任务复杂度

### 2. 设置合理超时

| Agent 类型 | 建议超时 |
|------------|----------|
| 产品 Agent | 4 小时 |
| UI设计 Agent | 8 小时 |
| 前端 Agent | 8 小时 |
| 后端 Agent | 8 小时 |
| 测试 Agent | 4 小时 |

### 3. 监控执行状态

- 实时查看各 Agent 进度
- 及时发现和处理阻塞
- 记录执行日志

### 4. 结果及时汇总

- Agent 完成后立即收集结果
- 更新问题状态
- 生成进度报告

### 5. 迭代优化

- 分析执行效率
- 优化任务分配
- 改进调度策略
