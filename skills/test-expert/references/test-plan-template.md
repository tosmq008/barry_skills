## 完整测试方案结构

- 编写日期: [日期]
- 编写人: 测试专家

## 1. 测试目标

### 1.1 主要目标
- 验证需求树中的核心场景都被叶子用例覆盖
- 验证关键业务流程的外部可观察结果
- 验证异常路径与边界条件
- 验证浏览器主流程与接口契约
- 验证小白用户是否能从 0 开始独立完成首个关键任务
- 验证多个用户并行使用时生态链路是否能稳定串联并闭环

### 1.2 质量目标
| 指标 | 目标值 |
|------|--------|
| 实际执行通过率 | ≥ 95% |
| Critical Bug | 0 |
| Major Bug | ≤ 3 |
| P0 场景叶子执行率 | 100% |
| 小白用户验收结论 | 不低于“需引导后可完成” |
| 多用户生态验收结论 | 不低于“需约束条件后可闭环” |

---

## 2. 测试范围

### 2.1 需求树范围
| 需求节点 | 能力节点 | 核心场景 | 风险等级 | 优先级 |
|----------|----------|----------|----------|--------|
| REQ_AUTH | CAP_LOGIN | SCN_LOGIN_SUCCESS / SCN_LOGIN_INVALID_PASSWORD | High | P0 |
| REQ_ORDER | CAP_SUBMIT | SCN_SUBMIT_SUCCESS / SCN_SUBMIT_DUPLICATE | High | P0 |
| REQ_ONBOARDING | CAP_FIRST_USE | SCN_NOVICE_FIRST_LOGIN_AND_CREATE | High | P0 |
| REQ_MARKETPLACE | CAP_ORDER_MATCH | SCN_ECO_MULTI_USER_ORDER_FLOW | High | P0 |

### 2.2 执行路径范围
| execution_route | 覆盖对象 | 是否承担黑盒主验证 | 工具 | 预计叶子数 |
|-----------------|----------|--------------------|------|------------|
| UNIT | 函数、模块级逻辑 | 否 | pytest / jest | [n] |
| INT_API | 服务 / API / 数据流 | 是 | pytest + requests/httpx/supertest | [n] |
| INT_UI | 浏览器主流程 | 是 | Playwright | [n] |

### 2.3 不在本轮范围
- 性能测试
- 安全渗透测试
- 压力测试
- 非主流浏览器兼容性

## 4. 测试策略

### 4.1 树状设计策略
- 先按需求树建模，再设计叶子
- 每个场景节点至少拆出 1 个叶子用例
- 功能黑盒叶子必须映射到 `INT_API` 或 `INT_UI`
- 禁止建立独立“黑盒测试章节”
- 首次使用主链路必须补充至少 1 个 `SCN_NOVICE_*` 场景用于验收
- 多边生态主链路必须补充至少 1 个 `SCN_ECO_*` 场景用于验收

### 4.2 执行策略
- 先做环境验证
- 再做每条执行路径的冒烟测试
- 再执行叶子用例
- 最后按场景树回写覆盖结果
- 对 `SCN_NOVICE_*` 场景，执行后必须输出单独的验收结论
- 对 `SCN_ECO_*` 场景，执行后必须输出单独的生态验收结论

### 4.3 回归策略
- 历史失败叶子优先
- 历史 Bug 对应叶子强制回归
- P0 / P1 场景抽样复验

## 5. 测试数据

### 5.1 原则
- 独立性
- 可重复性
- 最小化
- 真实性

### 5.2 规则
| 数据类型 | 管理方式 |
|----------|----------|
| 基础数据 | 测试前初始化 |
| 叶子用例数据 | 用例内创建和清理 |
| 可变共享数据 | 禁止使用 |

## 6. 风险评估

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 需求变更导致场景树失真 | 中 | 高 | 先更新树，再补叶子 |
| 环境不可用导致集成路径阻塞 | 中 | 高 | 执行前完成环境验证 |
| 浏览器脚本不稳定 | 中 | 中 | 保留 trace，优先修复选择器与等待策略 |
