# 客户端自测规范标准

> 适用平台：iOS / Android / Web / Flutter / 微信小程序

---

## 一、自测规范总则

### 1.1 自测的目的与意义

自测是开发流程中不可或缺的质量关口，其核心目标：

- **降低缺陷流出率**：在提测前拦截 80% 以上的低级缺陷，减少 QA 打回次数
- **提升交付效率**：减少开发与测试之间的反复沟通成本，缩短迭代周期
- **培养质量意识**：让开发者对自己的代码质量负责，形成"谁开发、谁自测、谁负责"的文化
- **保障用户体验**：从开发阶段就关注交互细节、异常处理和性能表现

### 1.2 自测覆盖范围

| 维度 | 说明 | 优先级 |
|------|------|--------|
| 功能验证 | PRD 功能点逐条验证，正向 + 反向流程 | P0 |
| 交互验证 | 页面流转、状态展示、手势响应 | P0 |
| 数据验证 | 列表/详情/表单数据正确性与完整性 | P0 |
| 异常处理 | 网络异常、服务端异常、并发操作 | P1 |
| 性能验证 | 启动时间、页面加载、滚动流畅度、内存 | P1 |
| 兼容性验证 | 设备/系统/浏览器/基础库版本覆盖 | P1 |
| 安全验证 | 数据加密、HTTPS、日志脱敏、权限合规 | P2 |

### 1.3 自测时机

| 阶段 | 自测内容 | 产出物 |
|------|----------|--------|
| 开发中 | 单模块功能验证、基本交互验证 | 无（开发过程自检） |
| 提测前 | 全量功能自测 + 异常场景 + 兼容性 | 自测报告（必须） |
| 联调后 | 接口数据验证、端到端流程验证 | 联调确认记录 |
| 回归阶段 | Bug 修复验证 + 关联功能回归 | Bug 状态更新 |

> **硬性要求**：未提交自测报告的需求，QA 有权拒绝接收提测。

## 二、功能自测清单

### 2.1 PRD 功能点逐条验证

- [ ] 将 PRD 中的功能点拆解为可验证的测试项
- [ ] 每个功能点至少覆盖一条正向用例和一条反向用例
- [ ] 功能点与 UI 设计稿逐一比对，确认文案、布局、颜色一致
- [ ] 确认功能点的前置条件和后置影响

### 2.2 正向流程验证

- [ ] 主流程能够正常走通（从入口到完成）
- [ ] 各步骤之间的数据传递正确
- [ ] 操作成功后的反馈提示正确（Toast/跳转/刷新）
- [ ] 操作成功后相关页面数据同步更新
- [ ] 多步骤流程中，中间步骤的状态保持正确

### 2.3 反向流程验证

| 场景类型 | 验证要点 |
|----------|----------|
| 空输入 | 必填字段为空时的提示是否正确 |
| 非法输入 | 特殊字符、超长文本、SQL 注入字符串 |
| 边界值 | 最小值、最大值、临界值（如 0、-1、MAX_INT） |
| 格式校验 | 手机号、邮箱、身份证号等格式验证 |
| 重复操作 | 重复提交、重复创建是否有幂等保护 |
| 取消操作 | 流程中途取消，数据是否正确回滚 |

### 2.4 多角色/多权限验证

- [ ] 不同角色登录后，菜单/按钮/数据的可见性符合权限配置
- [ ] 无权限时的提示信息友好且准确
- [ ] 权限变更后（升级/降级），页面实时反映变化
- [ ] 接口层面的权限校验（不仅仅是前端隐藏）
- [ ] 游客/未登录状态下的功能降级处理

## 三、交互自测清单

### 3.1 页面流转正确性

- [ ] 所有页面跳转路径与设计稿一致
- [ ] 返回操作的页面栈管理正确（不出现页面堆积）
- [ ] 深层页面的返回能正确回到预期页面
- [ ] DeepLink / URL Scheme 跳转正确
- [ ] 页面间参数传递完整且正确

### 3.2 按钮/手势响应

- [ ] 所有可点击元素有明确的点击反馈（高亮/缩放/波纹）
- [ ] 按钮防重复点击（节流/禁用处理）
- [ ] 手势操作响应正确（滑动、长按、双击、捏合缩放）
- [ ] 点击热区大小合理（最小 44x44pt / 48x48dp）
- [ ] 键盘弹出/收起时页面布局自适应

### 3.3 加载状态

| 状态 | 验证要点 |
|------|----------|
| Loading | 首次加载时展示 Loading 指示器 |
| Skeleton | 骨架屏与实际内容布局一致 |
| 下拉刷新 | 刷新动画正确，数据更新后动画收起 |
| 上拉加载 | 加载更多时有 Loading 提示，无更多数据时提示"没有更多了" |
| 局部加载 | 局部操作（如点赞）不触发全页面刷新 |

### 3.4 空状态/错误状态展示

- [ ] 列表为空时展示空状态占位图和文案
- [ ] 搜索无结果时展示专属空状态
- [ ] 网络错误时展示错误状态页，提供重试按钮
- [ ] 服务端错误时展示友好的错误提示
- [ ] 权限不足时展示引导提示
- [ ] 空状态和错误状态的插图与文案符合设计规范

### 3.5 Toast/Dialog/BottomSheet 交互

- [ ] Toast 展示时长合理（一般 1.5~2 秒），不遮挡关键操作区域
- [ ] Dialog 的确认/取消按钮功能正确
- [ ] Dialog 点击遮罩层的关闭行为符合预期（可关闭/不可关闭）
- [ ] BottomSheet 支持手势下滑关闭
- [ ] 多个弹窗的层级和展示优先级正确
- [ ] 弹窗内容超长时支持滚动

## 四、数据自测清单

### 4.1 列表数据正确性

- [ ] 默认排序规则正确（时间倒序/权重排序等）
- [ ] 切换排序方式后数据重新排列
- [ ] 筛选条件生效，筛选结果准确
- [ ] 多条件组合筛选结果正确
- [ ] 分页加载数据不重复、不遗漏
- [ ] 分页边界：第一页、最后一页、空页
- [ ] 列表总数与实际数据条数一致

### 4.2 详情数据完整性

- [ ] 详情页所有字段与接口返回数据一致
- [ ] 富文本内容渲染正确（图片/链接/表格/代码块）
- [ ] 时间格式统一且时区正确
- [ ] 金额/数量等数值的精度和格式正确
- [ ] 长文本的截断和展开/收起功能正常
- [ ] 图片/视频/文件的预览和下载功能正常

### 4.3 表单提交与回显

- [ ] 表单各字段的默认值正确
- [ ] 编辑模式下，已有数据正确回填
- [ ] 提交后数据持久化成功（刷新页面数据不丢失）
- [ ] 表单校验规则与 PRD 一致
- [ ] 联动字段的逻辑正确（如省市区级联）
- [ ] 图片/文件上传成功后正确展示
- [ ] 草稿保存与恢复功能正常

### 4.4 缓存数据一致性

- [ ] 缓存数据与服务端数据保持同步
- [ ] 数据更新后缓存及时失效或刷新
- [ ] 切换账号后缓存数据正确隔离
- [ ] 清除缓存后应用功能不受影响
- [ ] 离线缓存数据在恢复网络后正确同步

## 五、异常场景自测

### 5.1 网络异常

| 场景 | 验证要点 | 测试方法 |
|------|----------|----------|
| 断网 | 展示断网提示，恢复后可重试 | 开启飞行模式 |
| 弱网 | 请求不超时卡死，有 Loading 提示 | Charles 限速（3G/2G） |
| 超时 | 超时后展示友好提示，支持重试 | Charles 设置延迟 > 30s |
| 网络切换 | WiFi/4G 切换时请求不中断 | 手动切换网络 |
| DNS 劫持 | HTTPS 证书校验，防止中间人攻击 | 抓包工具验证 |

### 5.2 服务端异常

- [ ] HTTP 400：展示参数错误提示
- [ ] HTTP 401/403：跳转登录页或展示无权限提示
- [ ] HTTP 404：展示资源不存在提示
- [ ] HTTP 500：展示服务器异常提示，不暴露技术细节
- [ ] HTTP 502/503：展示服务暂不可用提示，引导稍后重试
- [ ] 接口返回数据格式异常：不崩溃，展示兜底 UI
- [ ] 接口返回空数据：正确展示空状态

### 5.3 认证异常

- [ ] Token 过期后自动刷新或引导重新登录
- [ ] Refresh Token 过期后强制登录
- [ ] 多设备登录互踢场景处理正确
- [ ] 账号被禁用/注销后的提示与跳转
- [ ] 权限被回收后实时生效（不依赖缓存的旧权限）

### 5.4 并发操作

- [ ] 快速连续点击按钮不触发多次请求
- [ ] 表单重复提交有防护机制（前端禁用 + 后端幂等）
- [ ] 多个请求并发时 UI 状态不错乱
- [ ] 竞态条件处理：后发先至的请求结果不覆盖最新请求
- [ ] 同一数据的并发编辑有冲突提示

### 5.5 前后台切换

| 平台 | 验证要点 |
|------|----------|
| iOS | 进入后台再恢复，页面状态保持；长时间后台后恢复，Token 刷新正确 |
| Android | 后台被系统回收后恢复，数据不丢失；多任务切换不崩溃 |
| Web | 标签页切换回来后，长连接自动重连；页面可见性变化时的数据刷新 |
| Flutter | 各平台生命周期回调正确处理；后台音视频播放策略正确 |
| 小程序 | onHide/onShow 生命周期处理正确；后台超过一定时间被销毁后的恢复 |

### 5.6 内存不足

- [ ] 低内存警告时释放非必要资源（大图缓存、音视频缓冲）
- [ ] 内存不足时不发生 OOM 崩溃
- [ ] 图片加载使用合理的缩放和缓存策略
- [ ] 大列表使用虚拟化/复用机制，避免内存持续增长

## 六、性能自测

### 6.1 各平台性能基线标准

| 指标 | iOS | Android | Web | Flutter | 小程序 |
|------|-----|---------|-----|---------|--------|
| 冷启动时间 | < 2s | < 3s | - | < 3s | < 3s |
| 首屏渲染 | < 1s | < 1.5s | < 2s (FCP) | < 1.5s | < 2s |
| 页面跳转 | < 300ms | < 300ms | < 500ms | < 300ms | < 500ms |
| 列表 FPS | >= 55 | >= 50 | >= 55 | >= 55 | >= 50 |
| 内存占用 | < 200MB | < 250MB | < 150MB | < 200MB | < 128MB |
| 包体积 | < 50MB | < 40MB | < 2MB(首包) | < 50MB | < 2MB(主包) |

> 以上为通用基线，具体项目可根据业务复杂度适当调整，但需在自测报告中说明。

### 6.2 启动时间测量方法

| 平台 | 工具/方法 | 关键指标 |
|------|-----------|----------|
| iOS | Instruments > App Launch | pre-main + post-main 时间 |
| Android | adb shell am start -W | TotalTime 字段 |
| Web | Performance API / Lighthouse | TTFB + FCP + LCP |
| Flutter | DevTools > Timeline | 引擎初始化 + 首帧渲染 |
| 小程序 | 微信开发者工具 > 性能面板 | 启动耗时 + 首屏耗时 |

### 6.3 页面加载时间

- [ ] 首屏内容在基线时间内渲染完成
- [ ] 非首屏内容使用懒加载，不阻塞首屏
- [ ] 图片使用合适的格式和尺寸（WebP/AVIF 优先）
- [ ] 接口请求合理并行，避免串行瀑布流
- [ ] 关键资源预加载（Prefetch/Preload）

### 6.4 列表滚动流畅度

- [ ] 快速滚动时无明显卡顿（FPS 不低于基线）
- [ ] 大量数据（1000+ 条）滚动不掉帧
- [ ] 列表项中的图片异步加载，不阻塞滚动
- [ ] 滚动过程中无白屏/闪烁现象
- [ ] 测量工具：
  - iOS：Instruments > Core Animation FPS
  - Android：GPU 呈现模式分析 / Systrace
  - Web：Chrome DevTools > Performance > Frames
  - Flutter：DevTools > Performance Overlay

### 6.5 内存占用监控

- [ ] 应用启动后内存占用在基线范围内
- [ ] 反复进出页面，内存无持续增长（排查内存泄漏）
- [ ] 大图/视频页面退出后内存正确释放
- [ ] 长时间使用（30 分钟+）内存保持稳定
- [ ] 监控工具：
  - iOS：Instruments > Leaks / Allocations
  - Android：Android Studio Profiler > Memory
  - Web：Chrome DevTools > Memory > Heap Snapshot
  - Flutter：DevTools > Memory

### 6.6 包体积检查

- [ ] 包体积不超过基线标准
- [ ] 无冗余资源文件（未使用的图片/字体/库）
- [ ] 图片资源已压缩优化
- [ ] 代码混淆和 Tree Shaking 已启用
- [ ] 动态库/Framework 按需加载
- [ ] 小程序分包策略合理，主包不超限

## 八、安全自测

### 8.1 敏感数据加密存储

| 数据类型 | 存储要求 | 验证方法 |
|----------|----------|----------|
| 用户密码 | 不在客户端存储明文 | 检查本地存储/数据库 |
| Token | 使用 Keychain(iOS) / EncryptedSP(Android) | 检查存储位置和加密方式 |
| 用户隐私数据 | 加密存储，不明文写入日志 | 检查日志输出和本地文件 |
| 支付信息 | 不在客户端持久化 | 检查本地存储 |
| 缓存数据 | 退出登录时清除敏感缓存 | 退出后检查本地文件 |

### 8.2 网络请求安全

- [ ] 所有请求使用 HTTPS，禁止 HTTP 明文传输
- [ ] 证书校验开启（防止中间人攻击）
- [ ] 敏感参数不通过 URL Query 传递（使用 POST Body）
- [ ] 请求头中不携带多余的敏感信息
- [ ] API 签名/加密机制正确实现

### 8.3 日志安全

- [ ] Release 包中关闭 Debug 日志输出
- [ ] 日志中不包含用户密码、Token、手机号等敏感信息
- [ ] 崩溃日志上报中脱敏处理敏感字段
- [ ] 第三方 SDK 的日志级别已设置为 Release 级别
- [ ] 网络请求日志中不打印完整的请求/响应体

### 8.4 权限申请合规

| 平台 | 验证要点 |
|------|----------|
| iOS | Info.plist 中权限描述文案准确；仅在使用时申请权限；被拒绝后引导设置页 |
| Android | AndroidManifest 中权限声明最小化；运行时权限按需申请；适配 Android 13+ 细化权限 |
| Web | 仅在用户操作触发时申请权限（摄像头/麦克风/定位）；权限被拒后的降级方案 |
| 小程序 | scope 权限按需申请；授权弹窗文案清晰；被拒后通过 wx.openSetting 引导 |

通用要求：
- [ ] 不申请与功能无关的权限
- [ ] 权限申请时机合理（使用时申请，非启动时全量申请）
- [ ] 权限被拒绝后功能优雅降级，不崩溃
- [ ] 隐私政策中声明的权限与实际申请一致

## 九、自测报告模板

### 9.1 报告格式

```markdown
# 自测报告

## 基本信息

| 项目 | 内容 |

### 9.3 遗留问题标注规范

遗留问题必须满足以下条件才可标注为"遗留"：

1. **已评估影响范围**：明确该问题影响的用户群体和使用场景
2. **已确认严重程度**：P0 问题不允许遗留，P1 问题需技术负责人审批
3. **已制定修复计划**：明确修复版本和负责人
4. **已告知相关方**：产品经理和 QA 负责人知悉并同意

遗留问题标注格式：

```
[遗留-P2] 问题描述
- 影响范围：仅影响 Android 8.0 以下设备（用户占比 < 2%）
- 原因说明：系统 API 不支持，需要额外适配
- 修复计划：v2.2.0 版本修复，负责人 [姓名]
- 审批人：[技术负责人姓名]
