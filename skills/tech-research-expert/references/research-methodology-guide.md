# 业界调研方法论指南

## 核心理念

> 调研的目的不是收集信息，而是**建立决策信心**。好的调研让团队快速对齐信息认知，减少盲决策。

## 调研信息源分级

### Tier 1: 高可信度来源（优先搜索）
| 来源类型 | 示例 | 可信度 | 注意事项 |
|----------|------|--------|---------|
| 官方文档 | 技术框架/工具的官方文档 | ★★★★★ | 注意版本差异 |
| 一线大厂技术博客 | Google/Meta/Uber/字节/阿里 Engineering Blog | ★★★★☆ | 规模差异可能很大 |
| 权威技术会议分享 | QCon/ArchSummit/KubeCon/VLDB 演讲 | ★★★★☆ | 可能有Marketing成分 |
| 有影响力的开源项目 | GitHub Stars > 5K、活跃维护 | ★★★★☆ | 评估维护状态 |

### Tier 2: 中可信度来源（交叉验证）
| 来源类型 | 示例 | 可信度 | 注意事项 |
|----------|------|--------|---------|
| 技术书籍 | O'Reilly 系列、Martin Fowler 等 | ★★★★☆ | 可能滞后1-2年 |
| 中等规模公司技术博客 | Medium 技术文章、个人技术博客 | ★★★☆☆ | 需交叉验证 |
| 学术论文 | ACM/IEEE 论文、arXiv 预印本 | ★★★☆☆ | 理论≠实践 |
| 行业报告 | Gartner/ThoughtWorks 技术雷达 | ★★★☆☆ | 可能有商业倾向 |

### Tier 3: 参考性来源（补充视角）
| 来源类型 | 示例 | 可信度 | 注意事项 |
|----------|------|--------|---------|
| 社区讨论 | Stack Overflow/Reddit/HN | ★★☆☆☆ | 观点≠事实 |
| 教程/课程 | Udemy/YouTube 教程 | ★★☆☆☆ | 简化版不适用生产环境 |
| 供应商材料 | 产品白皮书/宣传材料 | ★☆☆☆☆ | 高度偏向性 |

## 搜索关键词构造策略

### 中文关键词
```
基础: {技术问题} + 方案/架构/实践/优化
大厂: {公司名} + {技术问题} + 架构/实战/经验
规模: {技术问题} + 千万级/亿级/高并发/大规模
对比: {方案A} vs {方案B} + 对比/选型/区别
```

### 英文关键词
```
基础: {tech problem} + solution/architecture/best practices
大厂: {company} + {tech topic} + at scale / engineering
规模: {tech problem} + high throughput / low latency / scalable
对比: {solution A} vs {solution B} + comparison/benchmark/tradeoffs
前沿: {tech topic} + state of the art / 2024 / latest
```

### 跨界搜索策略
```
Step 1: 将问题抽象到更高层次
  例: "消息队列选型" → "高吞吐异步通信系统选型"
  
Step 2: 在其他行业搜索相同抽象问题
  例: 金融行业的交易消息系统、游戏行业的实时通信系统
  
Step 3: 借鉴核心思路，评估适配性
```

## 调研深度控制

| 调研层级 | 时间投入 | 产出深度 | 适用场景 |
|----------|---------|---------|---------|
| **快扫** | 1-2 小时 | 知道有哪些方向、各方向的大致特点 | 初步排除不可行方向 |
| **中度调研** | 0.5-1 天 | 理解各方案的核心思路、优缺点、适用场景 | 常规技术选型 |
| **深度调研** | 2-5 天 | 深入理解方案细节、复现 Benchmark、阅读源码 | 关键技术决策 |
| **POC 验证** | 1-2 周 | 原型实现、真实环境验证 | 高风险/高代价决策 |

## 调研信息记录规范

### 方案卡片记录法

每个调研到的方案，必须用标准化卡片记录：

```markdown
### 方案: [方案名称]
- **来源**: [公司/项目名 + 文章标题 + URL]
- **发布时间**: [YYYY-MM]
- **问题背景**: [他们面对什么问题？什么规模？]
- **核心方案**: [他们怎么解决的？核心架构/算法/策略]
- **技术栈**: [使用了什么技术/语言/框架]
- **效果数据**: [达到了什么效果？有没有数据？]
- **代价分析**:
  - 人力投入: [X 人 × Y 月]
  - 技术复杂度: [高/中/低]
  - 基础设施: [需要什么额外基础设施]
  - 运维成本: [日常运维的工作量]
- **局限性**: [方案的局限和不适用场景]
- **适配性评估**:
  - 匹配点: [与我们场景一致的部分]
  - 差异点: [与我们场景不同的部分]
  - 可借鉴: [可以直接借鉴的部分]
  - 需改造: [需要针对性改造的部分]
```

## 常见调研陷阱

1. **幸存者偏差**：只看到成功案例，忽视了失败案例和沉默大多数。
2. **规模谬误**：Google/Meta 的方案不一定适合百万级用户的产品。
3. **时间滞后**：2年前的最佳实践可能已经被更好的方案替代。
4. **过度搜索**：在已有充足信息后继续搜索，属于决策回避行为。
5. **确认偏见**：只搜索支持自己预设观点的证据，忽视反面证据。
