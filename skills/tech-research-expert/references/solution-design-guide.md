# 候选方案设计指南

## 核心理念

> 好的方案设计不是凭空创造，而是基于充分调研的"站在巨人肩膀上"的再创造。核心在于结合自身约束做有理有据的取舍。

## 方案设计原则

### 1. 杠杆原则

**找到投入产出比最高的切入点：**

```
杠杆公式：
  杠杆效果 = (问题解决程度) / (资源投入)

最佳切入点满足：
  ✅ 解决了 80% 的问题（Pareto 法则）
  ✅ 占用 20% 的资源
  ✅ 风险可控（失败后果可承受）
  ✅ 快速见效（可以尽早验证假设）
```

### 2. 排除法原则（资源有限时）

```
排除法决策树：
├── Step 1: 排除不满足硬约束的方案
│   └── 技术栈不兼容？时间不够？成本超标？→ 直接排除
│
├── Step 2: 排除代价远大于收益的方案
│   └── ROI < 1？维护成本远超开发成本？→ 排除
│
├── Step 3: 排除风险不可控的方案
│   └── 依赖不成熟技术？团队完全没经验？→ 排除（除非杠杆极大）
│
└── Step 4: 在剩余方案中选择杠杆最大的
    └── 比较 杠杆效果 = 解决程度 / 资源投入
```

### 3. 并行验证原则（资源充足时）

```
并行验证策略：
├── 设计 2-3 个差异化方案
├── 定义统一的度量标准（公平比较的前提）
├── 在相同条件下并行实施
├── 用 AB Test 框架收集数据
├── 基于数据做最终决策
└── 允许"组合最优"：从 A 方案取一部分 + 从 B 方案取一部分
```

## 方案差异化设计策略

设计多个候选方案时，确保方案之间有本质差异：

| 差异化策略 | 说明 | 示例 |
|-----------|------|------|
| **架构路线差异** | 不同的架构模式 | 微服务 vs 模块化单体 |
| **技术栈差异** | 不同的核心技术 | Redis vs Memcached |
| **复杂度梯度** | 从简到繁的方案层次 | 简单版 → 标准版 → 旗舰版 |
| **取舍方向差异** | 在不同目标间做不同取舍 | 性能优先 vs 成本优先 |
| **时间维度差异** | 短期快赢 vs 长期最优 | 临时方案 vs 终态方案 |

## 方案设计质量检查

- [ ] 每个方案是否有业界调研作为支撑（不是拍脑袋设计）？
- [ ] 方案之间是否有本质差异（而非同一思路的微调）？
- [ ] 优缺点分析是否坦诚（不隐瞒缺点）？
- [ ] 代价估算是否包含了隐性成本？
- [ ] 是否明确了每个方案的适用条件和失效条件？
- [ ] 实现路径是否足够具体，让开发团队可以评估可行性？
