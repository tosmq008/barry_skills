# 调研分析框架指南

## 核心理念

> 调研不是抄作业。从"信息收集"到"分析洞察"的跃迁，是调研产生价值的关键环节。

## 分析维度框架

### 1. 技术路线归类法

将调研到的多个方案，按底层技术路线进行归类：

```
归类原则：
├── 按核心技术差异分类（如：关系型 vs NoSQL vs NewSQL）
├── 按架构模式差异分类（如：集中式 vs 分布式 vs Serverless）
├── 按解决思路差异分类（如：空间换时间 vs 算法优化 vs 硬件升级）
└── 跨界方案单独归类
```

### 2. 代价-收益分析矩阵

**代价的完整维度：**

| 代价类型 | 子维度 | 评估方法 |
|----------|--------|---------|
| **开发成本** | 人力投入（人月） | 按功能拆解估算 |
| | 学习曲线 | 团队已有技能 vs 需新学习 |
| | 原型验证成本 | POC 的时间投入 |
| **运维成本** | 日常维护工作量 | 需要多少 SRE 时间 |
| | 监控/告警成本 | 需要建设什么监控 |
| | 故障处理复杂度 | 排查链路长度、恢复时间 |
| **基础设施成本** | 服务器/云资源 | 资源需求估算 |
| | 中间件/工具 | 是否需要引入新组件 |
| | 许可证费用 | 商业软件费用 |
| **机会成本** | 团队时间占用 | 占用团队多少产能 |
| | 技术债务 | 引入方案可能带来的技术债 |
| | 锁定风险 | 对特定技术/供应商的绑定 |

**收益的量化方法：**

| 收益类型 | 量化方式 | 示例 |
|----------|---------|------|
| 性能提升 | 前后对比数据 | 延迟从 500ms 降到 50ms |
| 成本节省 | 资源对比 | 服务器从 100 台降到 30 台 |
| 效率提升 | 操作对比 | 部署从 2 小时降到 5 分钟 |
| 稳定性 | 可用率 | 从 99.9% 提升到 99.99% |
| 可维护性 | 代码指标 | 模块耦合度降低、测试覆盖率提升 |

### 4. 适配性分析框架

**三圈分析法：**

```
    ┌──────────────────┐
    │   业界方案的       │
    │   前提条件集合     │
    │                    │
    │    ┌──────────┐   │
    │    │ 匹配区域 │   │    ← 可以直接借鉴的部分
    │    │          │   │
    └────┤          ├───┘
         │          │
    ┌────┤          ├───┐
    │    │          │   │
    │    └──────────┘   │
    │                    │
    │   我们场景的       │
    │   实际条件集合     │
    └──────────────────┘
    
差异区域分析：
├── 业界有、我们没有的条件 → 需要评估是否可以补齐，或方案需要适配
├── 我们有、业界没有的条件 → 可能是差异化优势，或额外约束
└── 重叠区域 → 可以直接复用的部分
```

**适配性评估清单：**

| 评估项 | 业界方案前提 | 我们的实际情况 | 差距 | 适配策略 |
|--------|------------|---------------|------|---------|
| 数据规模 | | | | |
| 团队规模 | | | | |
| 技术栈 | | | | |
| 基础设施 | | | | |
| 时间要求 | | | | |
| 可用预算 | | | | |

## 分析质量检查

- [ ] 是否避免了"抄方案"而是做了适配性分析？
- [ ] 每个方案的代价是否包含了隐性成本（运维、学习曲线、技术债）？
- [ ] 关键发现是否来自数据/事实而非主观判断？
- [ ] 适配性分析是否考虑了自身场景的特殊性？
- [ ] 是否有至少一个反面证据/反对意见的记录？
