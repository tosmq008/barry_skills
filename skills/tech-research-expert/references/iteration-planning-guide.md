# 迭代规划指南

## 核心理念

> "没有一次性完美的方案，只有持续迭代逼近最优的过程。每次迭代的价值不仅在于改进方案，更在于积累经验和减少不确定性。"

## 迭代规划原则

### 1. 迭代而非瀑布

```
❌ 瀑布思维：一次设计完美方案 → 大规模实施 → 祈祷成功
✅ 迭代思维：小步验证 → 获取数据 → 调整方向 → 再次验证 → 逐步扩大
```

### 2. 每次迭代的三个产出

| 产出 | 说明 | 作用 |
|------|------|------|
| **技术产出** | 方案的改进/调优 | 推进问题的解决 |
| **数据产出** | 度量指标的变化 | 验证假设、支撑决策 |
| **经验产出** | 团队的学习和沉淀 | 降低后续迭代的风险 |

### 3. 迭代退出条件

```
退出条件设计：
├── 成功退出: [核心指标达到目标值且稳定运行 X 天]
├── 方向转换: [连续 N 次迭代核心指标无改善 → 评估是否换方向]
├── 止损退出: [投入已超过预算阈值且收益不明显 → 停止投入]
└── 升级退出: [问题性质发生变化，需要更高层级的方案]
```

## 里程碑设计模板

### 里程碑结构

```markdown

[本迭代要达成的具体目标，用 SMART 原则描述]

- 前置条件: [需要什么准备就绪]
- 上一迭代经验: [M[N-1] 的关键学习]
- 数据基线: [当前指标的基线值]

### 执行内容
| 任务 | 负责人 | 时间 | 产出 |
|------|--------|------|------|
| [任务1] | | [X天] | |
| [任务2] | | [X天] | |

### 核心指标
| 指标 | 基线值 | 本迭代目标 | 度量方式 |
|------|--------|-----------|---------|
| [指标1] | [值] | [目标] | [方式] |

### 回顾要点

- [ ] 核心指标达到 [目标值]

```

### 典型的迭代节奏

```
M0: 基线建立（Week 0-1）
├── 建立度量体系
├── 采集基线数据
├── 确认调研结论和实施方案
└── 退出条件: 度量系统搭建完成 + 基线数据已记录

M1: 核心验证（Week 2-4）
├── 实施推荐方案的核心模块
├── 小范围验证效果
├── 收集第一批数据
└── 退出条件: 核心模块上线 + 小范围数据符合预期

M2: 效果调优（Week 5-8）
├── 基于 M1 数据调优参数/策略
├── 扩大验证范围
├── 处理边缘场景
└── 退出条件: 核心指标在更大范围达标

M3: 全量推广与稳定（Week 9-12）
├── 灰度推广至全量
├── 长期稳定性验证
├── 性能回归测试
├── 文档和知识沉淀
└── 退出条件: 全量上线 + 稳定运行 X 天 + 文档完善
```

## 信息对齐策略

### 信息对齐的时机

| 时机 | 对齐内容 | 形式 | 参与者 |
|------|---------|------|--------|
| 调研完成 | 调研发现 + 调研结论 | 调研报告评审会 | 全部决策相关方 |
| 方案确定 | 候选方案 + 推荐方案 | 方案评审会 | 全部决策相关方 |
| 每个 Sprint | 进展 + 数据 + 问题 | 站会/周会 | 执行团队 |
| 里程碑完成 | 里程碑报告 | 里程碑评审会 | 全部相关方 |
| 项目结束 | 沉淀文档 | 技术分享 | 更广泛受众 |

### 信息对齐的原则

1. **前置调研信息**：在方案评审会上，先花 30% 时间分享调研发现，让所有人对齐信息基础。
2. **结论先行**：每次分享先说结论和推荐，再说理由和过程。
3. **数据说话**：用数据而非观点说服人。
4. **记录异议**：记录所有不同意见，作为后续验证的参考。
5. **对齐 ≠ 共识**：目标是确保所有人了解同样的信息，而非强求意见统一。

## 迭代常见陷阱

1. **镀金效应**：方案已经达标但仍然继续优化。→ 设定明确的退出条件。
2. **沉没成本**：因前期投入太多而不愿换方向。→ 每个里程碑都重新评估 Go/No-Go。
3. **回顾敷衍**：Sprint 回顾流于形式，不产生实际调整。→ 回顾必须有具体的 Action Item。
4. **经验黑洞**：经验存在个人脑中不分享。→ 强制写入迭代文档和沉淀文档。
5. **指标漂移**：中途改变度量标准导致前后不可比。→ 指标变更需要记录原因并重新基线。
