# 度量与实验设计指南

## 核心理念

> "如果你无法度量它，你就无法改进它。" —— Peter Drucker
> "度量的目的不是证明自己对了，而是在错了的时候尽早发现。"

## 度量指标设计

### 指标四层模型

```
┌─────────────────────────────────────────────────────────┐
│                     度量指标体系                          │
├─────────────┬─────────────┬──────────────┬──────────────┤
│  核心指标   │  辅助指标   │   护栏指标   │  效率指标    │
│             │             │              │              │
│ 直接衡量    │ 辅助判断    │ 确保优化     │ 衡量方案     │
│ 问题是否    │ 方案效果    │ 不引入       │ 的实施       │
│ 被解决      │ 的多面性    │ 新问题       │ 代价         │
│             │             │              │              │
│ 例: 接口    │ 例: CPU     │ 例: 错误     │ 例: 开发     │
│ P99 延迟    │ 使用率      │ 率不上升     │ 人天数       │
└─────────────┴─────────────┴──────────────┴──────────────┘
```

### 指标设计规范

| 要素 | 说明 | 示例 |
|------|------|------|
| **名称** | 指标的清晰命名 | "接口 P99 延迟" |
| **定义** | 指标的精确计算方式 | "99% 的请求响应时间 ≤ X ms" |
| **采集方式** | 如何获取数据 | "APM 系统 + Prometheus" |
| **采集频率** | 多久采集一次 | "每 10 秒" |
| **基线值** | 当前的值（变更前） | "P99 = 520ms" |
| **目标值** | 期望达到的值 | "P99 ≤ 100ms" |
| **告警阈值** | 什么情况触发告警 | "P99 > 200ms 持续 5 分钟" |
| **数据负责人** | 谁负责采集和校验 | "SRE 团队" |

### 好的度量指标 vs 坏的度量指标

| ✅ 好的指标 | ❌ 坏的指标 |
|------------|------------|
| P99 延迟 ≤ 100ms | "系统更快了" |
| 错误率 < 0.1% | "稳定性提高了" |
| 每日处理订单 > 100万 | "吞吐量增加了" |
| 部署频率 > 10次/天 | "部署更频繁了" |
| 平均恢复时间 < 5分钟 | "恢复更快了" |

## 实验设计

### AB Test 设计框架

```
AB Test 设计模板：

1. 实验背景
   - 问题: [要验证什么]
   - 假设: [预期方案 A 比 B 在 [指标] 上好 [X]%]
   
2. 实验分组
   - 对照组: [描述] (占比 X%)
   - 实验组: [描述] (占比 Y%)
   - 分流维度: [用户ID/地域/时间/流量百分比]
   - 分流方法: [哈希/随机/按规则]
   
3. 观测指标
   - 主指标: [直接衡量假设的指标]
   - 辅指标: [辅助判断效果]
   - 护栏指标: [确保不引入问题]
   
4. 样本量估算
   - 最小可检测效应 (MDE): [X%]
   - 统计显著水平: [α = 0.05]
   - 统计功效: [1-β = 0.8]
   - 计算所需样本量: [N 个]
   
5. 实验周期
   - 预热期: [X 天]（排除新鲜效应）
   - 观测期: [Y 天]（收集足够样本）
   - 全周期 ≥ 1 个完整业务周期（如7天覆盖周末效应）
   
6. 成功标准
   - 主指标: [提升 X% + 统计显著 p < 0.05]
   - 护栏指标: [不恶化超过 Y%]
   
7. 停止规则
   - 提前成功: [效果远超预期且统计显著]
   - 提前终止: [护栏指标严重恶化 / 线上事故]
   
8. 分析计划
   - 整体效果分析
   - 分群分析（按用户类型/地域/设备）
   - 长期效应分析（是否有衰减）
```

### 排除法验证设计（资源有限时）

```
排除法验证流程：

Phase 1: 桌面排除（0 成本）
├── 排除不满足硬约束的方案
├── 排除明显 ROI < 1 的方案
└── 产出: 从 N 个方案缩减到 2-3 个

Phase 2: 快速 POC（最小成本）
├── 时间: [1-3天]
├── 验证目标: [方案最大的风险假设]
├── POC 范围: [最小可验证的功能子集]
├── 评判标准:
│   ├── 技术可行: [能跑通核心链路吗]
│   ├── 性能可行: [简单压测，数量级是否达标]
│   └── 复杂度可控: [实现复杂度是否在预期内]
├── 排除条件: [什么情况直接排除]
└── 产出: 从 2-3 个方案确认 1-2 个可行方案

Phase 3: 小批量验证（可控成本）
├── 时间: [1-2周]
├── 范围: [X% 流量 / 特定用户群 / 特定场景]
├── 观测指标: [核心指标 + 护栏指标]
├── Go/No-Go 判断:
│   ├── Go: [核心指标达到目标 X% + 无严重问题]
│   └── No-Go: [核心指标未达标 / 出现严重问题]
└── 产出: 明确的一个推荐方案

Phase 4: 灰度推广（逐步扩大）
├── 阶段: 10% → 30% → 50% → 100%
├── 每阶段观察期: [X 天]
├── 每阶段检查:
│   ├── 核心指标是否持续达标
│   ├── 护栏指标是否稳定
│   └── 是否有边缘场景问题
├── 回滚条件: [什么情况下回滚到上一阶段]
└── 产出: 全量上线
```

## Benchmark 设计指南

### 微基准测试 (Micro-benchmark)

```
目的: 在隔离环境中比较具体技术组件的性能

设计要点:
├── 使用标准化的测试工具（如 JMH/wrk/ab/sysbench）
├── 控制变量: 相同硬件、相同数据集、相同网络环境
├── 预热: 排除冷启动影响（至少 5 分钟预热）
├── 多次运行: 至少 3 次取平均（排除偶发因素）
├── 记录环境信息: OS/CPU/内存/磁盘/网络/软件版本
└── 注意事项: 微基准测试结果 ≠ 生产环境结果
```

### 端到端压力测试

```
目的: 在接近生产的环境中验证方案的整体表现

设计要点:
├── 测试环境: 尽可能接近生产（配置/数据量/网络拓扑）
├── 测试数据: 使用脱敏后的真实数据（不用随机数据）
├── 流量模型: 模拟真实的请求分布（不均匀、有波峰波谷）
├── 观测指标: 延迟分布/吞吐量/错误率/资源使用率
├── 极限测试: 逐步加压找到瓶颈点和崩溃点
└── 持续时间: 至少 30 分钟稳态测试（排除短期效应）
```

## 度量常见陷阱

1. **Goodhart's Law**: 当一个度量成为目标时，它就不再是一个好的度量。→ 需要护栏指标防止"刷指标"。
2. **平均值陷阱**: 平均延迟看起来很好，但长尾可能很差。→ 必须看 P99/P999。
3. **新鲜感效应**: 刚上线时用户因为新鲜感导致数据偏高。→ 实验需要有足够的预热期。
4. **辛普森悖论**: 分群看都变好了，总体反而变差了（或反之）。→ 需要分整体+分群的数据。
5. **因果倒置**: 相关性 ≠ 因果性。→ 需要控制变量或使用 AB Test。
