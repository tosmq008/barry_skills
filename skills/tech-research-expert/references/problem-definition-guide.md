# 问题定义指南

## 核心理念

> "一个被良好定义的问题，已经解决了一半。" —— Charles Kettering

问题定义的质量直接决定了调研的方向和效率。模糊的问题定义会导致无效的调研、偏离目标的方案设计。

## 问题定义五步法

### Step 1: 区分症状与根因

| 层级 | 说明 | 示例 |
|------|------|------|
| **症状** | 直接观察到的表象 | "接口响应变慢了" |
| **直接原因** | 导致症状的直接技术原因 | "数据库查询耗时增加" |
| **根本原因** | 问题产生的根本原因 | "表无索引 + 数据量从10万增长到1000万" |

**5 Whys 分析法：**
1. 为什么接口慢？→ 数据库查询慢
2. 为什么查询慢？→ 全表扫描
3. 为什么全表扫描？→ 缺少合适的索引
4. 为什么缺索引？→ 设计阶段没有考虑数据增长
5. 为什么没考虑？→ 缺少容量规划流程

### Step 2: 量化问题

**量化模板：**
```
问题: [问题描述]
影响面:
  - 受影响用户数: [N 人/占比 X%]
  - 受影响场景: [场景列表]
  - 频率: [每秒/每天/每周 发生 N 次]
当前指标:
  - [指标1]: [当前值] (期望值: [X])
  - [指标2]: [当前值] (期望值: [X])
业务影响:
  - 用户体验: [描述]
  - 商业损失: [估算]
```

### Step 3: 界定边界

| 维度 | 在范围内 (In Scope) | 不在范围内 (Out of Scope) |
|------|---------------------|--------------------------|
| 功能 | | |
| 系统 | | |
| 时间 | | |
| 用户 | | |

### Step 4: 梳理约束

```
硬约束 (不可违反):
├── [约束1]
├── [约束2]
└── [约束3]

软约束 (可协商):
├── [约束1] → 协商空间: [描述]
├── [约束2] → 协商空间: [描述]
└── [约束3] → 协商空间: [描述]
```

### Step 5: 定义成功标准

**SMART 标准：**
| 标准 | 内容 |
|------|------|
| Specific (具体) | 解决什么具体问题 |
| Measurable (可度量) | 用什么指标衡量 |
| Achievable (可实现) | 在约束条件下是否可达 |
| Relevant (相关) | 与业务目标的关联 |
| Time-bound (有时限) | 什么时候要看到效果 |

## 常见陷阱

1. **X-Y 问题**：用户描述的是他尝试的解法(X)的问题，而不是原始问题(Y)。必须追问原始需求。
2. **过早收窄**：在问题定义阶段就假定了某种解法，导致调研视野受限。
3. **问题膨胀**：把一个具体问题上升到一个宏大的命题，导致调研失焦。
4. **缺少量化**：用"慢""多""大"等模糊词代替具体数字。
