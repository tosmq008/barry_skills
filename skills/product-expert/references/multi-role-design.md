# 多角色设计指南

## 概述

本文档定义如何设计多角色系统，包括 Client端（用户侧）、Admin端（管理侧）、运营后台的完整设计规范。

---

## 一、三端架构概述

### 1.1 三端定位

```
┌─────────────────────────────────────────────────────────────────┐
│                        三端架构定位                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────┐                                           │
│  │    Client端     │  面向：终端用户                            │
│  │    (用户侧)     │  目标：提供核心产品价值                     │
│  │                 │  特点：体验优先、简洁易用                   │
│  └─────────────────┘                                           │
│           │                                                     │
│           │ 数据流                                              │
│           ▼                                                     │
│  ┌─────────────────┐                                           │
│  │    Admin端      │  面向：管理员、客服                        │
│  │    (管理侧)     │  目标：管理用户和内容                       │
│  │                 │  特点：功能完整、效率优先                   │
│  └─────────────────┘                                           │
│           │                                                     │
│           │ 数据流                                              │
│           ▼                                                     │
│  ┌─────────────────┐                                           │
│  │    运营后台     │  面向：运营人员                            │
│  │    (运营侧)     │  目标：数据分析、活动运营                   │
│  │                 │  特点：数据驱动、灵活配置                   │
│  └─────────────────┘                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 三端差异对比

| 维度 | Client端 | Admin端 | 运营后台 |
|------|----------|---------|----------|
| 用户群体 | 终端用户 | 管理员/客服 | 运营人员 |
| 使用频率 | 高频 | 中频 | 中低频 |
| 核心诉求 | 体验好、效率高 | 功能全、操作快 | 数据准、配置灵活 |
| 设计风格 | 品牌化、情感化 | 专业、高效 | 数据化、工具化 |
| 信息密度 | 低 | 中高 | 高 |
| 交互复杂度 | 低 | 中 | 中高 |

## 二、角色权限设计

### 2.1 角色层级

```
┌─────────────────────────────────────────────────────────────────┐
│                        角色层级结构                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  超级管理员 (Super Admin)                                       │
│       │                                                         │
│       ├── 系统管理员 (System Admin)                             │
│       │       │                                                 │
│       │       ├── 运营管理员 (Operation Admin)                  │
│       │       │       │                                         │
│       │       │       └── 运营人员 (Operator)                   │
│       │       │                                                 │
│       │       └── 内容管理员 (Content Admin)                    │
│       │               │                                         │
│       │               └── 内容审核员 (Content Reviewer)         │
│       │                                                         │
│       └── 客服主管 (CS Manager)                                 │
│               │                                                 │
│               └── 客服人员 (CS Staff)                           │
│                                                                 │
│  ─────────────────────────────────────────────────────────────  │
│                                                                 │
│  终端用户 (End User)                                            │
│       │                                                         │
│       ├── VIP用户 (VIP User)                                    │
│       │                                                         │
│       └── 普通用户 (Normal User)                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 权限矩阵

**Admin端权限矩阵：**

| 功能模块 | 超级管理员 | 系统管理员 | 运营管理员 | 内容管理员 | 客服主管 | 客服人员 |
|----------|------------|------------|------------|------------|----------|----------|
| 用户管理 | ✅ 全部 | ✅ 全部 | 👁 查看 | ❌ | 👁 查看 | 👁 查看 |
| 内容管理 | ✅ 全部 | ✅ 全部 | ✅ 全部 | ✅ 全部 | ❌ | ❌ |
| 订单管理 | ✅ 全部 | ✅ 全部 | ✅ 全部 | ❌ | ✅ 处理 | ✅ 处理 |
| 数据统计 | ✅ 全部 | ✅ 全部 | ✅ 全部 | 👁 部分 | 👁 部分 | ❌ |
| 系统配置 | ✅ 全部 | ✅ 全部 | ❌ | ❌ | ❌ | ❌ |
| 权限管理 | ✅ 全部 | 👁 查看 | ❌ | ❌ | ❌ | ❌ |

图例：✅ 完全权限 | 👁 只读权限 | ❌ 无权限

### 2.3 权限设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| 最小权限 | 只授予完成工作所需的最小权限 | 客服只能查看用户信息，不能修改 |
| 职责分离 | 关键操作需要多人协作 | 退款需要客服申请+主管审批 |
| 操作留痕 | 所有敏感操作记录日志 | 记录谁在什么时间做了什么 |
| 权限继承 | 上级角色继承下级权限 | 管理员拥有运营人员的所有权限 |

## 三、Client端设计规范

### 3.1 设计原则

| 原则 | 说明 |
|------|------|
| 用户优先 | 一切设计以用户体验为中心 |
| 简洁直观 | 减少认知负担，降低学习成本 |
| 情感化 | 通过设计传递品牌情感 |
| 一致性 | 保持视觉和交互的一致性 |

### 3.2 页面结构

**标准页面结构：**
```
┌─────────────────────────────────────────┐
│              状态栏 (系统)               │
├─────────────────────────────────────────┤
│              导航栏 (44-56px)            │
│  ← 返回    标题    操作按钮              │
├─────────────────────────────────────────┤
│                                         │
│                                         │
│              内容区域                    │
│           (可滚动区域)                   │
│                                         │
│                                         │
├─────────────────────────────────────────┤
│              底部操作栏 (可选)            │
│         主要按钮 | 次要按钮              │
├─────────────────────────────────────────┤
│              底部导航 (49-56px)          │
│    首页  |  发现  |  消息  |  我的       │
└─────────────────────────────────────────┘
```

### 3.3 必须页面清单

| 页面类型 | 页面 | 必须状态 |
|----------|------|----------|
| 启动 | 启动页/引导页 | 品牌展示、新功能引导 |
| 认证 | 登录/注册/忘记密码 | 表单、验证、错误 |
| 首页 | 首页 | 有数据、空、加载中 |
| 列表 | 列表页 | 有数据、空、加载、筛选 |
| 详情 | 详情页 | 完整信息、操作区 |
| 表单 | 表单页 | 输入、校验、提交 |
| 结果 | 结果页 | 成功、失败、处理中 |
| 个人 | 个人中心 | 信息展示、功能入口 |
| 设置 | 设置页 | 配置项、开关 |
| 消息 | 消息列表 | 有消息、空、未读标记 |

### 3.4 视觉风格

| 元素 | 规范 |
|------|------|
| 配色 | 品牌主色为主，辅以中性色 |
| 圆角 | 较大圆角(8-16px)，柔和友好 |
| 阴影 | 轻微阴影，层次分明 |
| 图标 | 线性图标为主，风格统一 |
| 插画 | 品牌插画，情感化表达 |

## 四、Admin端设计规范

### 4.1 设计原则

| 原则 | 说明 |
|------|------|
| 效率优先 | 减少操作步骤，提高工作效率 |
| 信息密度 | 合理的信息密度，一屏展示更多 |
| 专业感 | 专业、可靠的视觉感受 |
| 可扩展 | 支持功能扩展和定制 |

### 4.3 必须页面清单

| 页面类型 | 页面 | 设计要点 |
|----------|------|----------|
| 登录 | 登录页 | 简洁、安全感 |
| 首页 | Dashboard | 核心指标、快捷入口 |
| 列表 | 数据列表 | 表格、筛选、批量操作 |
| 详情 | 数据详情 | 完整信息、操作记录 |
| 表单 | 新增/编辑 | 字段完整、校验清晰 |
| 配置 | 系统配置 | 分类清晰、说明完整 |

### 4.4 表格设计规范

**表格必备功能：**
| 功能 | 说明 |
|------|------|
| 分页 | 支持分页和每页条数设置 |
| 排序 | 支持列排序 |
| 筛选 | 支持多条件筛选 |
| 搜索 | 支持关键词搜索 |
| 批量操作 | 支持多选和批量操作 |
| 列配置 | 支持显示/隐藏列 |
| 导出 | 支持数据导出 |

**表格列类型：**
| 列类型 | 设计要点 |
|--------|----------|
| ID列 | 固定宽度，可复制 |
| 文本列 | 超长截断，hover显示全部 |
| 状态列 | 使用标签/徽章 |
| 时间列 | 统一格式，支持排序 |
| 操作列 | 固定右侧，常用操作外露 |

### 4.5 视觉风格

| 元素 | 规范 |
|------|------|
| 配色 | 中性色为主，功能色点缀 |
| 圆角 | 较小圆角(4-8px)，专业感 |
| 阴影 | 轻微阴影，区分层级 |
| 图标 | 线性图标，功能明确 |
| 字体 | 清晰易读，层级分明 |

## 六、三端协同设计

### 6.1 数据流转

```
用户操作 (Client) → 数据存储 → 管理查看 (Admin) → 运营分析 (运营后台)
                         ↑                              │
                         └──────── 运营配置 ────────────┘
```

### 6.2 状态同步

| 场景 | Client显示 | Admin显示 | 运营后台显示 |
|------|------------|-----------|--------------|
| 订单创建 | 订单详情 | 新订单提醒 | 订单数+1 |
| 内容审核 | 审核中 | 待审核列表 | 待审核数 |
| 活动上线 | 活动入口 | 活动状态 | 活动数据 |

### 6.3 设计一致性

**跨端一致性要素：**
| 要素 | 一致性要求 |
|------|------------|
| 术语 | 同一概念使用相同术语 |
| 状态 | 状态名称和颜色一致 |
| 数据 | 同一数据展示一致 |
| 操作 | 相同操作逻辑一致 |

- [ ] 权限控制正确
- [ ] 数据展示准确
- [ ] 批量操作支持

### 运营后台检查
- [ ] 数据指标完整
- [ ] 配置灵活可用
- [ ] 效果可追踪
- [ ] 操作有记录
- [ ] 权限控制合理
