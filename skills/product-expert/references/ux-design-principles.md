# UX 设计原则指南

## 概述

本文档定义产品设计中的核心UX原则，指导交互设计和用户体验优化。

---

## 一、尼尔森十大可用性原则

### 1. 系统状态可见性 (Visibility of System Status)

**原则：** 系统应始终让用户知道正在发生什么，通过适当的反馈在合理时间内告知用户。

**设计应用：**
| 场景 | 设计方案 |
|------|----------|
| 页面加载 | 骨架屏、加载动画、进度条 |
| 操作执行 | 按钮状态变化、Toast提示 |
| 后台处理 | 进度指示、预计时间 |
| 网络状态 | 离线提示、同步状态 |

**示例：**
```
✅ 好的设计：上传文件时显示进度条和预计剩余时间
❌ 差的设计：点击上传后无任何反馈，用户不知道是否成功
```

### 2. 系统与现实世界匹配 (Match Between System and Real World)

**原则：** 使用用户熟悉的语言、词汇和概念，遵循现实世界的惯例。

**设计应用：**
| 方面 | 建议 |
|------|------|
| 术语 | 使用用户语言，避免技术术语 |
| 图标 | 使用通用隐喻（垃圾桶=删除） |
| 流程 | 符合现实操作顺序 |
| 反馈 | 使用自然语言描述 |

### 3. 用户控制与自由 (User Control and Freedom)

**原则：** 用户经常会误操作，需要明确的"紧急出口"来离开不想要的状态。

**设计应用：**
- 提供撤销/重做功能
- 支持取消正在进行的操作
- 提供返回/关闭按钮
- 危险操作前确认

**必须支持撤销的场景：**
| 操作类型 | 撤销方式 |
|----------|----------|
| 删除内容 | 软删除+恢复 |
| 发送消息 | 撤回功能 |
| 表单提交 | 编辑/修改 |
| 批量操作 | 批量撤销 |

### 4. 一致性与标准 (Consistency and Standards)

**原则：** 用户不应该怀疑不同的词语、情况或操作是否意味着同一件事。

**一致性层级：**
| 层级 | 说明 | 示例 |
|------|------|------|
| 内部一致 | 产品内部统一 | 所有确认按钮都在右侧 |
| 外部一致 | 符合平台规范 | iOS/Android设计规范 |
| 行业一致 | 符合行业惯例 | 电商购物车图标 |

### 5. 错误预防 (Error Prevention)

**原则：** 比好的错误信息更好的是精心设计以防止问题发生。

**预防策略：**
| 策略 | 实现方式 |
|------|----------|
| 约束输入 | 下拉选择代替自由输入 |
| 智能默认 | 预填合理默认值 |
| 实时校验 | 输入时即时反馈 |
| 确认机制 | 危险操作二次确认 |
| 禁用状态 | 不可用时禁用按钮 |

### 6. 识别而非回忆 (Recognition Rather Than Recall)

**原则：** 通过使对象、操作和选项可见来减少用户的记忆负担。

**设计应用：**
- 显示最近使用/历史记录
- 提供搜索建议和自动完成
- 使用图标+文字组合
- 保持上下文信息可见

### 7. 灵活性与效率 (Flexibility and Efficiency of Use)

**原则：** 加速器可以加快专家用户的交互，同时不影响新手用户。

**分层设计：**
| 用户类型 | 设计策略 |
|----------|----------|
| 新手用户 | 引导式操作、默认配置 |
| 普通用户 | 常用功能易达、快捷入口 |
| 专家用户 | 键盘快捷键、批量操作、高级设置 |

### 8. 美学与极简设计 (Aesthetic and Minimalist Design)

**原则：** 界面不应包含无关或很少需要的信息。

**设计原则：**
- 每个页面只有一个主要目标
- 移除不必要的元素
- 使用留白创造呼吸感
- 信息分层，渐进披露

### 9. 帮助用户识别、诊断和恢复错误 (Help Users Recognize, Diagnose, and Recover from Errors)

**原则：** 错误信息应该用简单语言表达，精确指出问题，并建设性地提出解决方案。

**错误信息设计：**
```
✅ 好的错误信息：
"密码需要至少8个字符，包含数字和字母。当前只有6个字符。"

❌ 差的错误信息：
"Error 400: Invalid input"
```

**错误信息模板：**
```
[发生了什么] + [为什么发生] + [如何解决]
```

### 10. 帮助与文档 (Help and Documentation)

**原则：** 最好系统不需要文档就能使用，但可能需要提供帮助和文档。

**帮助系统设计：**
| 类型 | 场景 | 形式 |
|------|------|------|
| 即时帮助 | 复杂功能旁 | Tooltip、问号图标 |
| 引导帮助 | 新功能、新用户 | 引导浮层、教程 |
| 搜索帮助 | 用户主动寻求 | 帮助中心、FAQ |
| 人工帮助 | 复杂问题 | 客服入口 |

## 二、交互设计原则

### 2.1 费茨定律 (Fitts's Law)

**原则：** 到达目标的时间与目标大小和距离有关。

**设计应用：**
- 重要按钮要足够大（最小44x44px）
- 常用功能放在易达位置
- 相关操作靠近放置
- 边缘和角落是高效位置

### 2.2 希克定律 (Hick's Law)

**原则：** 选项越多，做决定的时间越长。

**设计应用：**
- 减少选项数量（7±2原则）
- 分类组织选项
- 提供默认推荐
- 渐进式披露

### 2.3 米勒定律 (Miller's Law)

**原则：** 普通人短期记忆容量约为7±2个信息块。

**设计应用：**
- 导航项不超过7个
- 表单分步骤填写
- 信息分组展示
- 使用分块技术

### 2.4 接近性原则 (Law of Proximity)

**原则：** 相近的元素被认为是相关的。

**设计应用：**
- 相关功能靠近放置
- 使用间距区分分组
- 表单标签靠近输入框
- 操作按钮靠近操作对象

### 2.5 相似性原则 (Law of Similarity)

**原则：** 相似的元素被认为是相关的。

**设计应用：**
- 同类操作使用相同样式
- 不同状态使用不同颜色
- 可点击元素有统一特征
- 分类使用视觉区分

## 三、移动端设计原则

### 3.1 拇指热区

**热区分布：**
```
┌─────────────────────┐
│     困难区域        │  ← 需要双手或调整握姿
├─────────────────────┤
│     可达区域        │  ← 稍微伸展可达
├─────────────────────┤
│     舒适区域        │  ← 单手拇指自然可达
└─────────────────────┘
```

**设计建议：**
- 主要操作放在底部
- 导航栏考虑底部Tab
- 危险操作放在不易误触位置

### 3.2 触控目标尺寸

| 平台 | 最小尺寸 | 推荐尺寸 |
|------|----------|----------|
| iOS | 44x44pt | 48x48pt |
| Android | 48x48dp | 56x56dp |
| 通用 | 44x44px | 48x48px |

### 3.3 手势设计

**常用手势：**
| 手势 | 常见用途 | 注意事项 |
|------|----------|----------|
| 点击 | 选择、确认 | 提供点击反馈 |
| 长按 | 更多操作 | 需要提示可长按 |
| 滑动 | 切换、删除 | 提供视觉引导 |
| 下拉 | 刷新 | 显示刷新状态 |
| 双指缩放 | 放大缩小 | 图片/地图场景 |

## 四、表单设计原则

### 4.1 表单布局

**推荐布局：**
- 单列布局（移动端必须）
- 标签在输入框上方
- 相关字段分组
- 必填项明确标识

### 4.2 输入优化

| 优化点 | 实现方式 |
|--------|----------|
| 减少输入 | 选择代替输入、自动填充 |
| 智能键盘 | 数字键盘、邮箱键盘 |
| 输入掩码 | 手机号、银行卡格式化 |
| 实时校验 | 输入时即时反馈 |

### 4.3 错误处理

**错误提示原则：**
1. 即时反馈（输入时或失焦时）
2. 明确位置（靠近错误字段）
3. 清晰说明（具体问题和解决方案）
4. 视觉突出（红色边框、图标）

## 五、反馈设计原则

### 5.1 反馈类型

| 类型 | 场景 | 形式 | 时长 |
|------|------|------|------|
| 即时反馈 | 点击、输入 | 状态变化 | 即时 |
| 操作反馈 | 提交、保存 | Toast | 2-3秒 |
| 结果反馈 | 成功、失败 | 结果页 | 用户关闭 |
| 系统反馈 | 通知、提醒 | 弹窗/推送 | 用户处理 |

### 5.2 加载状态设计

| 加载时长 | 设计方案 |
|----------|----------|
| < 1秒 | 无需特殊处理 |
| 1-3秒 | 加载动画/Spinner |
| 3-10秒 | 进度条+预计时间 |
| > 10秒 | 后台处理+通知 |

### 5.3 空状态设计

**空状态类型：**
| 类型 | 设计要点 |
|------|----------|
| 首次使用 | 引导用户开始 |
| 无数据 | 说明原因+引导操作 |
| 无结果 | 建议调整条件 |
| 错误状态 | 说明问题+重试 |

## 六、可访问性设计

### 6.1 WCAG 基本原则

| 原则 | 说明 | 设计要点 |
|------|------|----------|
| 可感知 | 信息可被感知 | 文字替代、对比度 |
| 可操作 | 界面可操作 | 键盘可达、足够时间 |
| 可理解 | 内容可理解 | 清晰语言、一致性 |
| 健壮性 | 兼容辅助技术 | 语义化、标准遵循 |

### 6.2 颜色对比度

| 元素类型 | 最小对比度 |
|----------|------------|
| 正文文字 | 4.5:1 |
| 大号文字 | 3:1 |
| 图标/图形 | 3:1 |

### 6.3 无障碍检查清单

- [ ] 所有图片有替代文字
- [ ] 颜色不是唯一的信息传达方式
- [ ] 表单有清晰的标签
- [ ] 错误信息可被屏幕阅读器读取
- [ ] 可通过键盘完成所有操作
- [ ] 焦点状态清晰可见

## 七、设计评审检查清单

### 交互设计检查

- [ ] 所有可点击元素有点击反馈
- [ ] 加载状态有明确指示
- [ ] 错误状态有友好提示
- [ ] 空状态有引导操作
- [ ] 危险操作有确认机制
- [ ] 支持撤销/返回操作

### 可用性检查

- [ ] 主要功能3步内可达
- [ ] 表单填写流程顺畅
- [ ] 信息层级清晰
- [ ] 操作反馈及时
- [ ] 帮助信息易获取

### 一致性检查

- [ ] 视觉风格统一
- [ ] 交互模式一致
- [ ] 术语使用统一
- [ ] 图标含义一致
- [ ] 反馈方式一致
