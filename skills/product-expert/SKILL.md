# 产品专家 Skill

作为全栈产品专家，融合商业分析、产品设计、UX/UED、用户增长等多领域专业能力，从0到1完成产品设计全流程。

## When to Use

**适用场景：**
- 新产品从0到1的完整设计
- 现有产品的功能迭代设计
- 产品重构和体验优化
- 竞品分析和产品对标
- 高保真原型和UI设计稿制作
- 详细PRD文档输出
- 多角色系统设计（Client/Admin/运营后台）
- 融入运营策略的产品设计
- 用户增长导向的功能设计

**不适用：**
- 纯技术架构设计（使用 tech-plan-template）
- 纯代码实现（使用开发相关 skill）
- 纯测试执行（使用 test-expert）

## Phase 1: 需求洞察 (Requirement Insight)

### 1.1 用户痛点挖掘

**痛点分析框架：**

| 维度 | 分析要点 | 输出 |
|------|----------|------|
| 功能痛点 | 现有方案无法满足的需求 | 功能需求清单 |
| 效率痛点 | 流程繁琐、操作复杂 | 效率优化点 |
| 体验痛点 | 界面混乱、反馈不及时 | 体验改进点 |
| 成本痛点 | 时间/金钱/学习成本高 | 成本优化方向 |
| 情感痛点 | 焦虑、不信任、挫败感 | 情感设计要点 |

### 1.2 用户场景分析

**场景描述模板：**
```
场景名称: [场景名]
用户角色: [谁]
触发条件: [什么情况下]
用户目标: [想要达成什么]
当前方案: [现在怎么做]
痛点问题: [遇到什么问题]
期望体验: [理想状态是什么]
```

### 1.3 商业目标定义

**SMART 目标框架：**

| 要素 | 说明 | 示例 |
|------|------|------|
| Specific | 具体明确 | 提升用户注册转化率 |
| Measurable | 可衡量 | 从 15% 提升到 25% |
| Achievable | 可实现 | 基于行业基准可达成 |
| Relevant | 相关性 | 与业务增长目标一致 |
| Time-bound | 有时限 | 3个月内达成 |

## Phase 5: UI设计 (UI Design with Pencil MCP)

> ⚠️ **执行前必须读取 `references/ui-design-workflow.md` 获取UI设计工作流**

### 5.1 设计准备

**设计前置工作：**
1. 确定设计风格（参考竞品和品牌调性）
2. 定义设计规范（颜色、字体、间距、圆角）
3. 准备设计素材（图标、图片、插画）
4. 明确设计约束（平台规范、技术限制）

### 5.2 使用 Pencil MCP 设计

**设计流程：**

```
Step 1: 获取设计灵感
├── 调用 get_style_guide_tags 获取可用标签
├── 调用 get_style_guide 获取风格指南
└── 确定设计方向

Step 2: 创建设计文件
├── 调用 open_document 创建/打开 .pen 文件
├── 调用 get_guidelines 获取设计规范
└── 调用 batch_get 了解设计系统组件

Step 3: 设计页面
├── 调用 batch_design 创建页面结构
├── 使用组件库快速搭建
└── 逐步完善细节

Step 4: 设计验证
├── 调用 get_screenshot 截图检查
├── 调用 snapshot_layout 检查布局
└── 迭代优化
```

## 用户增长方法论 (User Growth Methodology)

> ⚠️ **执行前必须读取 `references/user-growth-methodology.md` 获取用户增长方法论**

### AARRR模型在产品设计中的应用

| 阶段 | 目标 | 产品设计要点 | 关键指标 |
|------|------|--------------|----------|
| Acquisition | 获取用户 | 落地页优化、注册流程简化 | 注册转化率 |

## Best Practices

### 产品设计最佳实践

1. **用户中心** - 始终从用户视角出发，而非功能堆砌
2. **数据驱动** - 用数据验证假设，而非主观臆断
3. **迭代思维** - MVP先行，快速验证，持续优化
4. **全局视角** - 考虑多角色、多场景、多状态

### UI设计最佳实践

1. **一致性** - 保持视觉和交互的一致性
2. **层次感** - 信息有主次，视觉有层级
3. **可用性** - 易学易用，减少认知负担
4. **可访问性** - 考虑不同用户群体的需求

### PRD输出最佳实践

1. **完整性** - 覆盖所有场景和状态
2. **清晰性** - 描述准确，无歧义
3. **可执行** - 开发可直接依据实现
4. **可验收** - 有明确的验收标准
