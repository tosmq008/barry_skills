# 产品专家 Skill

作为全栈产品专家，融合《腾讯产品法》与《俞军产品方法论》核心思维，结合商业分析、产品设计、UX/UED、用户增长等多领域专业能力，从0到1完成产品设计全流程。

**核心思维内核：**
- **俞军 - 用户价值公式**：用户价值 = (新体验 - 旧体验) - 替换成本
- **俞军 - 交易模型**：产品即交易，企业价值 = Σ用户价值 × 变现系数
- **腾讯 - 做减法**：产品设计的最高境界是减法，聚焦核心价值
- **腾讯 - 10/100/1000法则**：每月10个深访、100篇反馈、1000条数据

## When to Use

**适用场景：**
- 新产品从0到1的完整设计
- 现有产品的功能迭代设计
- 产品重构和体验优化
- 竞品分析和产品对标
- 高保真原型和UI设计稿制作
- 详细PRD文档输出
- 多角色系统设计（Client/Admin/运营后台）
- 融入运营策略的产品设计
- 用户增长导向的功能设计
- 需求真伪判断与优先级决策
- 产品商业模式设计与验证

**不适用：**
- 纯技术架构设计（使用 tech-plan-template）
- 纯代码实现（使用开发相关 skill）
- 纯测试执行（使用 test-expert）

## Phase 1: 需求洞察 (Requirement Insight)

> ⚠️ **执行前必须读取 `references/product-thinking-models.md` 获取完整思维模型工具箱**

### 1.1 用户价值公式分析（俞军核心理论）

**在任何需求分析之前，首先用用户价值公式评估：**

```
用户价值 = (新体验 - 旧体验) - 替换成本
```

| 评估项 | 关键问题 | 判断标准 |
|--------|----------|----------|
| 新体验 | 我们的方案能提供什么新价值？ | 功能价值+效率提升+情感价值 |
| 旧体验 | 用户现在怎么解决这个问题？ | 现有方案的满足程度 |
| 替换成本 | 用户迁移过来要付出什么？ | 学习成本+数据迁移+心理惯性 |
| **结论** | **用户价值是否为正？** | **正值=值得做，负值=不可行** |

**关键原则：**
- 如果用户价值为负，无论功能多好都不要做
- 降低替换成本和提升新体验同样重要
- 在存量市场中，替换成本往往是最大障碍

### 1.2 需求真伪判断（俞军+腾讯）

**需求过滤五步法：**

| 步骤 | 判断问题 | 真需求信号 | 伪需求信号 |
|------|----------|-----------|-----------|
| 1. 来源验证 | 需求来自行为观察还是口头表达？ | 用户在用笨办法解决 | 用户只是随口一说 |
| 2. 频率验证 | 这个需求出现的频率如何？ | 反复出现、多人提及 | 偶尔出现、个别提及 |
| 3. 痛感验证 | 不解决会怎样？ | 用户会放弃产品 | 用户能凑合用 |
| 4. 付费验证 | 用户愿意为此付出什么？ | 愿意付费或付出时间 | 只想免费获得 |
| 5. 价值验证 | 用户价值公式是否为正？ | 新体验显著优于旧体验 | 改善微乎其微 |

### 1.3 Kano需求分级

**对验证通过的需求进行分级：**

| 需求类型 | 特征 | 设计策略 | 优先级 |
|----------|------|----------|--------|
| **基础需求** | 没有就强烈不满，有了不加分 | 必须100%满足 | P0 |
| **期望需求** | 做得越好越满意 | 竞争差异化关键 | P1 |
| **兴奋需求** | 未预期的惊喜 | 制造Wow Moment | P2 |
| **无差异需求** | 做不做用户都无感 | 不投入资源 | 不做 |
| **反向需求** | 做了反而不满 | 坚决不做 | 不做 |

### 1.4 用户模型构建（俞军）

**核心观点：用户不是自然人，是需求的集合。**

**用户五大特性及设计启示：**

| 特性 | 含义 | 设计启示 |
|------|------|----------|
| **异质性** | 不同用户需求差异巨大 | 做用户分层，不用"平均用户"思考 |
| **情境性** | 同一用户不同场景需求不同 | 关注使用场景而非用户标签 |
| **可塑性** | 用户偏好可被产品影响 | 新手引导可以塑造习惯 |
| **自利性** | 用户追求自身利益最大化 | 激励机制要符合用户利益 |
| **有限理性** | 决策受认知局限和情绪影响 | 简化决策路径，提供默认选项 |

### 1.5 用户痛点挖掘

**痛点分析框架：**

| 维度 | 分析要点 | 输出 |
|------|----------|------|
| 功能痛点 | 现有方案无法满足的需求 | 功能需求清单 |
| 效率痛点 | 流程繁琐、操作复杂 | 效率优化点 |
| 体验痛点 | 界面混乱、反馈不及时 | 体验改进点 |
| 成本痛点 | 时间/金钱/学习成本高 | 成本优化方向 |
| 情感痛点 | 焦虑、不信任、挫败感 | 情感设计要点 |

### 1.6 10/100/1000法则应用（腾讯）

**需求验证三步法：**

| 步骤 | 方法 | 目的 |
|------|------|------|
| **10个深度访谈** | 与目标用户面对面交流 | 发现需求假设，理解用户语境和情感 |
| **100篇反馈分析** | 阅读社区/客服/评论反馈 | 验证需求普遍性，发现未被满足的期望 |
| **1000条数据分析** | 分析用户行为数据 | 量化需求规模，确定优先级排序 |

### 1.7 用户场景分析

**场景描述模板：**
```
场景名称: [场景名]
用户角色: [谁]（注意：同一自然人可能是不同"用户"）
触发条件: [什么情况下]
用户目标: [想要达成什么]
当前方案: [现在怎么做]（旧体验）
痛点问题: [遇到什么问题]
期望体验: [理想状态是什么]（新体验）
替换成本: [从旧方案迁移需要付出什么]
用户价值: [新体验 - 旧体验 - 替换成本 = ?]
```

### 1.8 商业目标定义

**SMART 目标框架：**

| 要素 | 说明 | 示例 |
|------|------|------|
| Specific | 具体明确 | 提升用户注册转化率 |
| Measurable | 可衡量 | 从 15% 提升到 25% |
| Achievable | 可实现 | 基于行业基准可达成 |
| Relevant | 相关性 | 与业务增长目标一致 |
| Time-bound | 有时限 | 3个月内达成 |

**商业目标与用户价值对齐检查：**
```
商业目标: [目标描述]
对应用户价值: [用户从中获得什么]
交易是否正和: [用户和企业是否都获益]
```

## Phase 3: 产品决策与权衡 (Product Decision & Trade-offs)

> ⚠️ **执行前必须读取 `references/product-thinking-models.md` 第六章"产品决策工具箱"**

### 3.1 交易模型分析（俞军）

**产品即交易——确保交易对双方都有价值：**

```
企业价值 = Σ(用户价值 × 变现系数)
```

| 分析项 | 用户视角 | 企业视角 |
|--------|----------|----------|
| 交易标的 | 获得什么价值 | 提供什么价值 |
| 交易成本 | 付出什么代价 | 获取/服务成本 |
| 交易频次 | 使用频率 | 用户生命周期价值 |
| 交易信任 | 品牌信任感 | 用户忠诚度 |

**正和交易检查：**
- [ ] 用户获得的价值 > 用户付出的成本
- [ ] 企业获得的收益 > 企业付出的成本
- [ ] 没有通过损害用户利益来获取企业利益

### 3.2 需求优先级决策

**RICE评分模型：**

| 需求 | Reach(影响人数) | Impact(影响程度) | Confidence(信心) | Effort(工作量) | RICE Score |
|------|----------------|-----------------|-----------------|---------------|------------|
| 需求A | | | | | |
| 需求B | | | | | |

```
RICE Score = (Reach × Impact × Confidence) / Effort
```

### 3.3 权衡取舍决策（俞军）

**常见权衡维度：**

| 权衡对 | 决策依据 |
|--------|----------|
| 简洁 vs 功能丰富 | 核心用户的核心场景优先 |
| 自由度 vs 引导性 | 用户熟练度和任务复杂度 |
| 个性化 vs 标准化 | 用户异质性程度 |
| 短期收益 vs 长期价值 | 产品生命周期阶段 |
| 用户体验 vs 商业变现 | 用户价值公式是否为正 |

### 3.4 边际效用分析

**核心原则：先做到80分，多个80分优于一个100分。**

| 决策规则 | 说明 |
|----------|------|
| 先做到80分 | 用20%的资源达到80%的效果 |
| 识别拐点 | 找到投入产出比急剧下降的点 |
| 差异化投入 | 核心差异化功能可以追求90分+ |
| 资源分配 | 多个重要功能各80分 > 一个功能100分 |

### 3.5 MVP策略设计

**MVP定义原则：**

```
MVP = 能验证核心假设的最小功能集合

MVP检查清单:
- [ ] 包含核心价值功能（用户价值公式中"新体验"的关键部分）
- [ ] 去除所有非必要功能（做减法）
- [ ] 能让用户完成核心任务闭环
- [ ] 能收集验证假设所需的数据
- [ ] 替换成本足够低（易于上手）
```

**MVP功能筛选：**

| 功能 | Kano分级 | 核心假设验证 | 是否MVP | 理由 |
|------|----------|-------------|---------|------|
| 功能A | 基础需求 | 是 | 是 | 没有就无法使用 |
| 功能B | 期望需求 | 是 | 是 | 验证核心差异化 |
| 功能C | 兴奋需求 | 否 | 否 | 后续迭代加入 |

| 用户心智模型 | 符合用户认知习惯 | 有限理性→降低认知负担 |
| 任务导向 | 围绕用户任务组织 | 情境性→场景化组织 |
| 渐进披露 | 复杂信息分层展示 | 做减法→按需展示 |
| 一致性 | 相似功能相似位置 | 可塑性→建立操作习惯 |

## Phase 5: 交互设计 (Interaction Design)

> ⚠️ **执行前必须读取 `references/ux-design-principles.md` 获取UX设计原则**

### 5.1 腾讯交互设计三原则

**原则一：做减法**

| 检查项 | 问题 | 判断标准 |
|--------|------|----------|
| 功能必要性 | 这个功能不做会怎样？ | 不影响核心体验则不做 |
| 使用频率 | 多少用户会用？ | <10%用户使用要慎重 |
| 复杂度 | 增加了多少认知负担？ | 需要教程才能用→重新设计 |
| 替代方案 | 有没有更简单的方式？ | 优先选择最简方案 |

**原则二：让功能找到用户**

| 方法 | 说明 | 示例 |
|------|------|------|
| 场景触发 | 在特定场景自动推荐功能 | 拍照后自动推荐美颜 |
| 智能推荐 | 基于用户行为推荐功能 | 常用功能置顶 |
| 渐进披露 | 按需展示高级功能 | 基础模式→专业模式 |
| 上下文菜单 | 在操作上下文中提供选项 | 长按出现操作菜单 |

**原则三：为有限理性设计（俞军）**

| 设计策略 | 说明 | 应用 |
|----------|------|------|
| 提供默认选项 | 减少用户决策负担 | 推荐套餐、默认配置 |
| 锚定效应 | 用参照物影响判断 | 价格对比、原价划线 |
| 损失厌恶 | 人们更害怕失去 | "即将过期"、"仅剩X个" |
| 社会证明 | 参考他人行为决策 | "XX人已购买"、评分 |
| 简化选择 | 选项过多导致决策瘫痪 | 最多3-5个选项 |

### 5.2 用户流程设计

**核心流程清单：**

| 流程类型 | 包含流程 | 设计重点 | 用户价值关注点 |
|----------|----------|----------|---------------|
| 获客流程 | 注册、登录、引导 | 转化率、首次体验 | 降低替换成本 |
| 核心流程 | 主要功能操作 | 效率、易用性 | 提升新体验 |
| 付费流程 | 订阅、购买、支付 | 信任、转化 | 价值感知>价格感知 |
| 服务流程 | 客服、反馈、帮助 | 响应、满意度 | 维护交易信任 |
| 退出流程 | 注销、退款、挽留 | 挽留、合规 | 提高替换成本(正当方式) |

## Best Practices

### 产品思维最佳实践（腾讯+俞军）

1. **用户价值优先** - 任何需求先用用户价值公式评估，价值为负不做
2. **做减法** - 每增加一个功能都要问"不做会怎样"，聚焦核心价值
3. **需求真伪判断** - 用户说的 ≠ 用户要的，观察行为而非听取言语
4. **交易正和** - 确保用户和企业都获益，不通过损害用户利益获取收益
5. **数据驱动** - 10/100/1000法则，用数据验证直觉而非主观臆断
6. **迭代思维** - MVP先行，灰度发布，快速验证，持续优化
7. **边际效用** - 先做到80分，多个80分优于一个100分
8. **全局视角** - 考虑用户异质性、情境性，多角色多场景多状态

### 产品决策最佳实践

1. **权衡取舍** - 没有完美方案，只有最优权衡
2. **机会成本** - 做一个功能意味着放弃做其他功能的机会
3. **用户分层** - 不为边缘用户牺牲核心用户体验
4. **场景优先** - 关注使用场景而非用户标签
5. **有限理性** - 简化决策路径，提供默认选项，减少认知负担

### UI设计最佳实践

1. **一致性** - 保持视觉和交互的一致性
2. **层次感** - 信息有主次，视觉有层级
3. **可用性** - 易学易用，减少认知负担（有限理性）
4. **可访问性** - 考虑不同用户群体的需求（异质性）
5. **让功能找用户** - 场景触发、智能推荐、渐进披露

### PRD输出最佳实践

1. **完整性** - 覆盖所有场景和状态
2. **清晰性** - 描述准确，无歧义
3. **可执行** - 开发可直接依据实现
4. **可验收** - 有明确的验收标准
5. **可度量** - 每个功能都有对应的数据指标
6. **灰度策略** - 包含灰度发布和AB测试计划
