# WORKFLOW.md - 开发工作流约定

> 定义快速原型开发的 4 阶段工作流程

---

## 工作流总览

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   Phase 1    │ →  │   Phase 2    │ →  │   Phase 3    │ →  │   Phase 4    │
│  Idea (30m)  │    │  UI (2-4h)   │    │  Code (1-2d) │    │  Ship (4-8h) │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
     ↓                    ↓                    ↓                    ↓
  8个PRD文档          5轮UI审查           三端代码实现          测试+发布
```

---

## Phase 1: 需求分析 (30 minutes)

### 输出清单

| # | 文档 | 核心内容 |
|---|------|----------|
| 1 | 项目简介 | 问题、用户、核心功能、成功标准 |
| 2 | 功能架构图 | Client + Admin + Website 三端 |
| 3 | 系统角色 | 角色定义 + 权限矩阵 |
| 4 | 模块划分 | 功能点 + 优先级 (P0/P1/P2) |
| 5 | 页面清单 | 页面编号 + 入口 + 角色 |
| 6 | 跳转关系 | 页面间跳转流程图 |
| 7 | 交互操作 | 每页操作 + 组件 |
| 8 | 视觉风格 | Client vs Admin 差异 |

### 必须回答的问题

- 解决什么问题？
- 目标用户是谁？
- 核心功能是什么？（最多 5 个）
- 成功标准是什么？

### 完成标准

- [ ] 8 个 PRD 文档全部创建
- [ ] 功能架构图覆盖三端
- [ ] 所有页面有编号
- [ ] 角色权限矩阵完整

---

## Phase 2: UI 设计 (2-4 hours)

### 5 轮迭代审查

```
Round 1 (商业分析师) → Round 2 (领域PM) → Round 3 (资深PM) → Round 4 (UED) → Round 5 (小白用户)
```

| Round | 角色 | Focus |
|-------|------|-------|
| 1 | 商业分析师 | 价值主张、转化路径 |
| 2 | 领域产品经理 | 业务流程、行业实践 |
| 3 | 资深产品经理 | 用户习惯、交互体验 |
| 4 | UED 设计专家 | 视觉层次、设计一致性 |
| 5 | 小白用户 | 易用性、直观性 |

### 必须覆盖的页面

**Client Side:**
| 阶段 | 页面 |
|------|------|
| 入口 | 启动页、引导页、登录、注册、忘记密码 |
| 首页 | 首页(有数据)、首页(空状态)、首页(加载中) |
| 核心 | 列表页、详情页、搜索页、筛选页 |
| 操作 | 表单页、确认页、结果页(成功/失败) |
| 个人 | 个人中心、编辑资料、修改密码 |
| 消息 | 消息列表、消息详情、空消息 |
| 设置 | 系统设置、关于我们、意见反馈 |
| 异常 | 空状态、加载中、网络错误、404 |
| 组件 | 弹窗、Toast、ActionSheet |

**Admin Side:**
| 阶段 | 页面 |
|------|------|
| 入口 | 登录页 |
| 首页 | Dashboard(数据概览) |
| 列表 | 数据列表(有数据/空/加载中) |
| 表单 | 新增表单、编辑表单 |
| 详情 | 数据详情页 |
| 配置 | 系统配置页 |
| 组件 | 确认弹窗、操作反馈 |

**Website:**
| 页面 | 功能 |
|------|------|
| Landing | Hero + 功能 + CTA |
| 功能介绍 | 产品功能详细说明 |
| 定价页面 | 价格方案 |
| 关于我们 | 团队/公司介绍 |
| 联系我们 | 联系方式、反馈表单 |
| 404 | 错误页面 |

### 完成标准

- [ ] 5 轮审查全部完成
- [ ] Client 所有页面状态覆盖
- [ ] Admin 所有页面覆盖
- [ ] Website 所有页面覆盖
- [ ] 设计稿保存到 `docs/ui/[project].pen`

## Phase 3: 代码实现 (1-2 days)

### 执行顺序

```
Setup (30m) → Client (1d) → Admin (0.5d) → Website (2-4h) → Integration (2-4h)
```

### 端口分配

| 服务 | 端口 |
|------|------|
| Client Frontend | 3000 |
| Client Backend | 8000 |
| Admin Frontend | 3001 |
| Admin Backend | 8001 |
| Website | 4000 |

### 任务清单 (10 个任务)

```
3.1 初始化项目结构和目录
3.2 实现 Client Backend 数据模型
3.3 实现 Client Backend API 接口
3.4 实现 Client Frontend 页面框架
3.5 实现 Client Frontend 核心页面
3.6 实现 Admin Backend 数据模型和 API
3.7 实现 Admin Frontend 页面
3.8 实现 Website 静态页面
3.9 使用 frontend-design skill 优化 Website
3.10 创建启动脚本 start.sh/stop.sh
```

### 核心规则

- ✅ 后端代码放 `src/` 包下
- ✅ 界面严格按设计稿还原
- ✅ Website 也必须先完成 UI 设计稿
- ✅ Website 必须用 `frontend-design` skill 优化
- ❌ 禁止脱离设计稿自行发挥
- ❌ 禁止跳过设计稿直接写代码

### 完成标准

- [ ] Client Frontend + Backend 完成
- [ ] Admin Frontend + Backend 完成
- [ ] Website 静态页面完成
- [ ] 启动脚本可用
- [ ] 所有服务可正常启动

## Phase 4: 测试发布 (4-8 hours)

### 测试范围

| 类型 | 必须？ | 说明 |
|------|--------|------|
| 单元测试 | ✅ | 核心逻辑 |
| 集成测试 | ✅ | API 层 |
| 功能测试 | ✅ | 必须 |
| E2E 测试 | ⚠️ | 核心流程 |
| 性能测试 | ❌ | 不包含 |
| 安全测试 | ❌ | 不包含 |

### 任务清单 (6 个任务)

```
4.1 编写 Client Backend 单元测试
4.2 编写 Admin Backend 单元测试
4.3 执行集成测试
4.4 执行功能测试
4.5 修复所有发现的 Bug
4.6 最终验收检查
```

### 测试用例独立性要求

- 每个用例使用独立的测试数据
- 测试数据在用例执行前创建，执行后清理
- 不依赖其他用例产生的数据
- 测试必须能任意次数重复执行

### 发布检查清单

- [ ] 核心功能正常工作
- [ ] 无阻塞性 Bug
- [ ] UI 与设计稿一致
- [ ] 基本错误处理
- [ ] 启动脚本可用

### 完成标准

- [ ] 所有测试通过
- [ ] Bug 已修复
- [ ] 验收检查通过
- [ ] 可交付使用

---

## 迭代周期

发布后快速迭代：

```
Feedback → Fix/Improve → Test → Ship → Repeat
  (1hr)      (2-4hrs)    (1hr)  (30min)
```
